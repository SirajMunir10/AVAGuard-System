**Definition**

Identity Protection is a feature in Microsoft Entra ID that uses machine learning and behavioral analytics to detect and respond to identity-based risks. It evaluates sign-in attempts and user behavior to assign risk levels (low, medium, high) for both users and sign-ins. Risk policies are Conditional Access policies that automatically respond to these risk detections, such as requiring MFA for medium-risk sign-ins or blocking high-risk users. Identity Protection helps organizations automate the detection and remediation of compromised accounts.

**Components**

Identity Protection consists of several components. Risk detections: events that indicate potential compromise, such as leaked credentials, impossible travel, anonymous IP address, unfamiliar sign-in properties, malware-linked IP, and password spray. User risk: the probability that a user's identity has been compromised, calculated based on all risk detections associated with that user. Sign-in risk: the probability that a specific sign-in attempt is unauthorized, based on real-time signals. Risk policies: two built-in Conditional Access policies—"User risk policy" and "Sign-in risk policy"—that you can configure to automatically respond. Remediation actions: when a risk is detected, you can require MFA, require password change, block access, or allow self-remediation via the user risk policy. Investigation tools: the "Risky users" and "Risky sign-ins" reports provide detailed information about each detection, including risk level, detection type, and date. Identity Protection also includes a "Risk detections" report for a historical view. Additionally, you can configure "MFA registration policy" to require users to register for MFA if they are at risk.

**Application**

To configure Identity Protection risk policies, navigate to Entra ID > Protection > Identity Protection. Start by reviewing the "Risky users" and "Risky sign-ins" reports to understand current risk levels. Then, under "User risk policy", click "Configure". Set the policy to target "All users" and assign a user risk level of "High". Under "Access", choose "Block access" or "Allow access with password change" (recommended: require password change). Enable the policy in "Report-only" mode first. Similarly, configure the "Sign-in risk policy": target "All users", sign-in risk level "Medium and above", and access control "Require multi-factor authentication". Enable in Report-only. After monitoring for a week, switch both policies to "On". For self-remediation, ensure users have registered MFA methods (use the Registration campaign). When a user is detected as high risk, they will be prompted to change their password and re-authenticate with MFA. For investigation, use the "Risky users" report to dismiss false positives (e.g., if a user's IP was incorrectly flagged as anonymous) by clicking "Dismiss user risk". You can also confirm compromise (if you believe the user is actually compromised) to trigger automatic remediation. For advanced scenarios, integrate with Microsoft Defender for Identity to correlate on-premises AD signals. Troubleshooting involves checking the "Risk detections" tab for each user to see the specific detection type and timestamp. Common issues include false positives from VPNs or shared IPs—you can add trusted IPs under Conditional Access named locations to reduce noise.

**Security Relevance**

Identity Protection automates the detection of compromised accounts, which is critical because attackers often use stolen credentials for weeks before detection. By automatically blocking high-risk sign-ins or requiring password changes, you can stop attacks in real time. The user risk policy is especially important: if a user's credentials are leaked on the dark web, Identity Protection will detect it and force a password change on next sign-in, preventing the attacker from using those credentials. The sign-in risk policy blocks impossible travel scenarios (e.g., a sign-in from New York and then from Tokyo within 5 minutes) which is a strong indicator of token theft. These policies reduce the burden on security teams by automating responses to common attack patterns. Compliance frameworks like NIST SP 800-63 require risk-based authentication, which Identity Protection directly provides. Without these policies, organizations rely on manual review of sign-in logs, which is slow and error-prone.

**Key Takeaways**
1. Identity Protection uses machine learning to detect risky users and sign-ins, enabling automated responses like MFA or password change.
2. Configure both user risk and sign-in risk policies, starting in Report-only mode to validate before enforcement.
3. These policies automate the detection and remediation of compromised accounts, reducing dwell time and manual effort.
