**Definition**

Multi-Factor Authentication (MFA) is a security mechanism that requires users to provide two or more verification factors to gain access to a resource. In Microsoft Entra ID, MFA adds a second layer of security beyond a password, typically using something you have (e.g., a phone, hardware token) or something you are (e.g., biometrics). Entra ID MFA supports several methods: Microsoft Authenticator app (push notification or OTP), SMS, voice call, hardware OATH tokens, and FIDO2 security keys. Enforcement can be per-user, per-application, or driven by Conditional Access policies.

**Components**

The MFA configuration in Entra ID involves several components. First, the MFA service itself is enabled at the tenant level. Second, authentication methods are managed under Protection > Authentication methods > Policies, where you can enable or disable specific methods (e.g., Microsoft Authenticator, SMS, voice call) and configure their usage (e.g., passwordless phone sign-in, OTP only). Third, per-user MFA state can be set to Disabled, Enabled, or Enforced via the legacy MFA blade (though Microsoft recommends using Conditional Access instead). Fourth, MFA registration campaign settings allow you to nudge users to register their methods. Fifth, the MFA service settings (under Protection > MFA > Additional cloud-based MFA settings) include options like app passwords (for legacy apps that don't support MFA) and trusted IPs (to bypass MFA from corporate locations). Finally, Conditional Access policies are the modern, granular way to enforce MFA based on conditions like user risk, location, or application sensitivity.

**Application**

To set up MFA enforcement using best practices, first ensure all users have registered at least one MFA method. Use the Registration campaign feature under Protection > Authentication methods > Registration campaign to send periodic prompts to unregistered users. Next, create a Conditional Access policy: navigate to Protection > Conditional Access > New policy. Name it "Require MFA for All Users". Under Assignments, select Users and groups > All users. Under Cloud apps or actions > All cloud apps. Under Conditions, you can optionally exclude trusted locations (e.g., corporate office IPs) by configuring Named locations under Protection > Conditional Access > Named locations. Under Access controls > Grant, select "Require multi-factor authentication". Set Enable policy to "Report-only" initially. Monitor sign-in logs to see how many users would be affected. After a week, switch to "On". For users who cannot use modern MFA methods (e.g., service accounts), exclude them from the policy and use other controls like managed identities or certificate-based authentication. Troubleshooting MFA issues involves checking user authentication method registration (under Users > [user] > Authentication methods), verifying that the user's MFA state is not set to Disabled in the legacy per-user MFA blade (which can override Conditional Access), and ensuring that the user's device time is synchronized (common cause of OTP failures). For users receiving repeated MFA prompts, review sign-in frequency settings in Conditional Access session controls.

**Security Relevance**

MFA is one of the most effective controls against credential theft. Microsoft research shows that MFA blocks over 99.9% of account compromise attacks. Without MFA, a single stolen password can give an attacker full access to email, files, and applications. Enforcing MFA via Conditional Access allows you to apply it contextually—for example, requiring MFA only for external access or high-risk sign-ins, reducing user friction while maintaining security. Per-user MFA enforcement is less flexible and can lead to user fatigue if applied globally without exceptions. Modern enforcement via Conditional Access also supports passwordless methods like FIDO2 keys and Windows Hello for Business, which are resistant to phishing. Compliance frameworks such as PCI DSS, HIPAA, and SOC 2 require MFA for remote access and privileged accounts. Additionally, MFA helps prevent lateral movement: even if an attacker compromises one account, they cannot easily access other resources without the second factor.

**Key Takeaways**
1. Enforce MFA using Conditional Access policies rather than per-user settings for granular, risk-based control.
2. Use the Registration campaign to drive user adoption and ensure all users have registered authentication methods.
3. MFA blocks over 99.9% of account compromise attacks and is a fundamental requirement for compliance.
