Azure Bastion is a Platform as a Service (PaaS) offering from Microsoft Azure that provides secure, seamless Remote Desktop Protocol (RDP) and Secure Shell (SSH) access to virtual machines (VMs) within a virtual network (VNet), without exposing those VMs to the public internet. It operates as a fully managed jump server, provisioned directly inside your VNet, and eliminates the need for public IP addresses on target VMs, bastion hosts, or complex VPN/ExpressRoute configurations. From a security perspective, Bastion mitigates the risk of port scanning, brute-force attacks, and lateral movement by ensuring all management traffic is routed through the Azure backbone and authenticated via Azure Active Directory (Azure AD) and role-based access control (RBAC).

**Key Components**

- **Azure Bastion Host Resource**: A dedicated PaaS instance deployed in a specific subnet named `AzureBastionSubnet` (minimum /27). This subnet must reside within the same VNet as the target VMs. The Bastion host is not a VM you manage; Azure handles patching, hardening, and scaling.
- **AzureBastionSubnet**: A required subnet with a route table that does not contain user-defined routes (UDRs) pointing to on-premises or virtual appliances for internet traffic. Network security groups (NSGs) on this subnet must allow inbound traffic from the Azure Gateway Manager service tag (for health probes) and outbound traffic to the target VMs on TCP 3389 (RDP) or TCP 22 (SSH).
- **Target VMs**: Must have private IP addresses only (no public IPs). They must be running a supported OS (Windows Server, Linux distributions) and have the appropriate RDP/SSH ports open only to the AzureBastionSubnet’s IP range. The VM’s NSG should restrict inbound RDP/SSH to the Bastion subnet’s address prefix.
- **Authentication and Authorization**: Bastion supports Azure AD authentication for Windows VMs (with Azure AD-joined or hybrid-joined VMs) and SSH key-based or password-based authentication for Linux VMs. Access is further controlled via Azure RBAC: users need the `Reader` role on the VM and the `Contributor` role on the Bastion resource (or a custom role allowing `Microsoft.Network/bastionHosts/connect/action`).
- **Session Recording and Monitoring**: Bastion integrates with Azure Monitor and Azure Activity Logs to record connection attempts, session durations, and authentication failures. For compliance, you can enable diagnostic settings to stream logs to Log Analytics workspaces or storage accounts.
- **Native Client Support**: Bastion offers browser-based access (HTML5) and native client support via the Azure CLI or the Azure portal’s “Connect” blade. Native client support uses a WebSocket tunnel over HTTPS, avoiding direct network exposure.

**Real-World Application**

Consider a financial services organization that hosts a critical application on Azure VMs in a production VNet. The security policy mandates no public IPs on any VM and requires all administrative access to be logged and audited. Without Bastion, the team would need to deploy a traditional jump box (a VM with a public IP) and manage its security patches, NSG rules, and VPN connectivity. This introduces attack surface and operational overhead.

With Bastion, the organization deploys a single Bastion instance in the production VNet’s AzureBastionSubnet. All administrators authenticate via Azure AD (using conditional access policies requiring multi-factor authentication). The target VMs have NSGs that only allow inbound RDP/SSH from the Bastion subnet’s IP range (e.g., 10.0.1.0/27). The security team enables diagnostic logging for all Bastion sessions, sending logs to a centralized SIEM. When an auditor requests a record of who accessed a specific VM and for how long, the team queries the Log Analytics workspace. This setup eliminates public exposure, reduces management overhead, and provides a clear audit trail.

Another common use case is hybrid scenarios where on-premises administrators need to manage Azure VMs without a site-to-site VPN. Bastion can be accessed over the internet via HTTPS (port 443), but the connection is authenticated and encrypted end-to-end. The target VM never sees an inbound connection from the public internet—only from the Bastion subnet. This is particularly useful for DevOps teams managing ephemeral test environments where VPN setup is impractical.

**Security Relevance**

- **Attack Surface Reduction**: By removing public IPs from VMs, you eliminate the most common vector for brute-force and vulnerability scanning attacks. Bastion itself is not directly accessible from the internet except on port 443 for the Azure portal or native client, and it is protected by Azure’s infrastructure-level DDoS protection and TLS 1.2+ encryption.
- **Just-in-Time (JIT) Access Integration**: Bastion can be combined with Azure Security Center’s JIT VM access. JIT temporarily opens the NSG rule for the Bastion subnet to allow a connection, then closes it after the session ends. This ensures that even the Bastion subnet’s access to VMs is time-bound and approved.
- **Privileged Identity Management (PIM)**: Because Bastion uses Azure AD authentication, you can enforce PIM policies for administrative roles. For example, a user must request elevation to the “Virtual Machine Administrator Login” role for a specific time window, and the request must be approved. This prevents standing admin access.
- **Session Isolation**: Each Bastion session is isolated in a separate browser tab or native client process. The Bastion service does not allow lateral movement between VMs; a user must explicitly connect to each VM. This limits the blast radius if credentials are compromised.
- **Compliance and Auditing**: Bastion logs include source IP, target VM, authentication method, session duration, and any errors. These logs are immutable if sent to a storage account with immutable blob storage. This satisfies regulatory requirements (e.g., PCI DSS, SOC 2) for monitoring administrative access.
- **Network Segmentation**: Bastion does not require any inbound ports on the target VMs from the internet. The only required network path is from the AzureBastionSubnet to the target VM’s private IP. This aligns with a zero-trust network model where no host trusts another host by default.

**Key Takeaways**
1. Azure Bastion eliminates public IP exposure on VMs by providing RDP/SSH access through a managed PaaS service inside your VNet, reducing attack surface and operational overhead.
2. Security is enforced through Azure AD authentication, RBAC, NSG restrictions on the AzureBastionSubnet, and integration with JIT and PIM for time-bound, approved access.
3. All sessions are logged and auditable via Azure Monitor, enabling compliance with regulatory standards and providing a clear forensic trail for incident response.
