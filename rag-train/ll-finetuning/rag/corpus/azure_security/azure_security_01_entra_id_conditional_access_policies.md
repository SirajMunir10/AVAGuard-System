**Definition**

Conditional Access is the core policy engine within Microsoft Entra ID (formerly Azure Active Directory) that enables you to enforce access controls based on specific conditions. Instead of a simple yes/no authentication decision, Conditional Access evaluates signals such as user identity, device state, location, application sensitivity, and real-time risk to determine whether to grant, block, or require additional verification for access to resources. It is a Zero Trust foundational component, ensuring that every access attempt is verified dynamically rather than trusted implicitly.

**Components**

A Conditional Access policy consists of three primary parts: assignments, conditions, and access controls. Assignments define who the policy applies to (users, groups, roles, or all users) and what cloud applications are targeted (e.g., Office 365, Azure Management, custom SaaS apps). Conditions add granularity by specifying sign-in risk, device platform (Windows, iOS, Android), location (IP ranges, countries, trusted networks), client apps (browser, mobile app, legacy authentication), and user risk. Access controls determine the outcome: block access, grant access with requirements (MFA, compliant device, hybrid Azure AD joined device, approved client app, or app protection policy), or session controls (sign-in frequency, persistent browser session, or app-enforced restrictions). Policies are evaluated in order, and if multiple policies apply, the most restrictive combination of controls is enforced.

**Application**

To configure a Conditional Access policy, navigate to the Microsoft Entra admin center under Protection > Conditional Access. Start by creating a new policy and naming it descriptively (e.g., "Block Legacy Auth for All Users"). Under Assignments, select Users and groups, then choose "All users" or a specific test group. Under Cloud apps or actions, select "All cloud apps" or a specific app like Exchange Online. Under Conditions, configure Client apps to block "Exchange ActiveSync clients" and "Other clients" (legacy protocols). Under Access controls, set Grant to "Block access". Enable the policy in "Report-only" mode first to test impact without enforcement, using the Conditional Access insights workbook to review sign-in logs. Once validated, switch to "On". For troubleshooting, use the What If tool (located in the Conditional Access blade) to simulate a sign-in attempt with specific user, app, and conditions to see which policies would apply and what controls would be enforced. Common issues include policy conflicts (multiple policies with contradictory grants), misconfigured locations (trusted IPs not matching actual egress IPs), and excluded users inadvertently left out of critical policies. Always monitor sign-in logs for "Failure" reasons like "Conditional Access policy blocked the request" and review the "Conditional Access" tab within each sign-in event for detailed policy evaluation results.

**Security Relevance**

Conditional Access is the primary mechanism to enforce least-privilege access and reduce the attack surface. By blocking legacy authentication protocols (which do not support MFA), you eliminate a common vector for credential stuffing and password spray attacks. Requiring MFA for all external access or high-risk sign-ins dramatically reduces account compromise risk. Device compliance policies ensure that only managed, healthy devices can access corporate data, preventing malware-laden personal devices from connecting. Location-based policies can block access from unexpected geographic regions or require MFA from untrusted networks. Session controls like sign-in frequency force reauthentication after a set time, limiting exposure from stolen tokens. Without Conditional Access, your Entra ID tenant relies solely on static passwords and basic MFA, leaving gaps that attackers exploit. Properly configured policies are essential for compliance frameworks like NIST, ISO 27001, and CMMC, which require dynamic access controls based on risk.

**Key Takeaways**
1. Conditional Access policies evaluate user, device, location, and risk signals to enforce dynamic access decisions, forming the backbone of Zero Trust in Entra ID.
2. Always test new policies in Report-only mode and use the What If tool to validate behavior before enabling enforcement.
3. Block legacy authentication and require MFA for high-risk scenarios to significantly reduce common attack vectors.
