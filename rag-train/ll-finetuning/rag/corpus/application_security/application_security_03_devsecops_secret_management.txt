Secret management is the discipline of securely storing, distributing, rotating, and auditing access to non-human credentials—API keys, database passwords, TLS certificates, service account tokens, and encryption keys—used by applications, infrastructure, and CI/CD pipelines. In a DevSecOps context, secret management replaces hard-coded secrets in source code, configuration files, or environment variables with dynamic, policy-controlled access to a centralized vault. The core objective is to eliminate secret sprawl, reduce the blast radius of a compromise, and enforce the principle of least privilege for machine-to-machine authentication.

**Key Components**

A robust secret management system comprises several interdependent components:

- **Centralized Vault**: A hardened, encrypted store (e.g., HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, CyberArk Conjur) that holds secrets at rest. The vault itself must be isolated from application workloads, often running on dedicated infrastructure with strict network policies. Encryption keys for the vault are typically managed via a hardware security module (HSM) or a cloud key management service (KMS).

- **Dynamic Secrets**: Instead of static, long-lived credentials, the vault generates short-lived, ephemeral secrets on demand. For example, a database credential might be valid for only 60 minutes. This limits the window of exposure if a secret is leaked. Dynamic secrets require the vault to integrate with the target service (e.g., a database engine or cloud provider API) to create and revoke credentials programmatically.

- **Policy-Based Access Control**: Access to secrets is governed by identity-based policies, not network location. Policies define which applications, services, or human users can read, write, or rotate specific secrets. Policies should be version-controlled and auditable. Common models include role-based access control (RBAC) and attribute-based access control (ABAC).

- **Secret Rotation**: Automated, scheduled rotation of secrets reduces the risk of credential reuse after a breach. Rotation can be manual (triggered by an operator) or automatic (triggered by a policy or time-based schedule). The vault must coordinate rotation with the target service to avoid downtime—for example, updating a database password while maintaining active connections.

- **Audit Logging**: Every read, write, and rotation operation on a secret must be logged with a timestamp, identity, and context. Logs feed into SIEM systems for anomaly detection and compliance reporting (e.g., SOC 2, PCI DSS). Audit trails are critical for post-incident forensics.

- **Encryption in Transit and at Rest**: Secrets are encrypted when stored in the vault (AES-256 or stronger) and when transmitted over the network (TLS 1.2+). The vault’s encryption keys must be separate from the vault data, ideally managed by an external KMS or HSM.

**Real-World Application**

Consider a microservices-based e-commerce platform deployed on Kubernetes. Without secret management, each service would have database passwords and API keys hard-coded in ConfigMaps or environment variables. This creates several risks: secrets are visible to anyone with cluster access, they are duplicated across namespaces, and rotation requires redeploying every service.

A practical implementation uses HashiCorp Vault with the Vault Agent Injector sidecar pattern. Each pod runs a Vault Agent sidecar that authenticates to the vault using the pod’s service account token (via Kubernetes auth method). The sidecar retrieves the required secrets at startup and writes them to a shared in-memory volume (e.g., `/vault/secrets`). The application reads secrets from this file, never from environment variables.

For database credentials, the application requests a dynamic PostgreSQL credential from Vault. Vault connects to the database, creates a user with a 24-hour TTL, and returns the username and password. The application uses these credentials for the session. After 24 hours, Vault automatically revokes the user. If a pod is compromised, the attacker gains access to a credential that expires within hours, not months.

For CI/CD pipelines, a Jenkins job authenticates to Vault using an AppRole role ID and secret ID. The pipeline retrieves a short-lived token to fetch deployment secrets (e.g., SSH keys for a production server). The token is valid only for the pipeline’s duration, preventing reuse.

**Security Relevance**

Secret management directly addresses several critical security concerns in DevSecOps:

- **Eliminates Hard-Coded Secrets**: Removing secrets from source code prevents accidental exposure via version control (e.g., committing a `.env` file to a public repo). It also stops secrets from appearing in logs, error messages, or build artifacts.

- **Reduces Blast Radius**: Dynamic secrets and short TTLs mean a single leaked credential cannot be used to pivot across systems. If a database password is compromised, it is valid for minutes, not months. Rotation policies further ensure that even if a secret is stolen, it becomes useless after the next rotation cycle.

- **Enforces Least Privilege for Machines**: Policies ensure that each service only has access to the secrets it needs. A payment processing service cannot read the database password for the user profile service. This containment limits lateral movement in a breach.

- **Provides Auditability**: Every secret access is logged. In a post-incident investigation, you can determine exactly which service, at what time, accessed which credential. This is essential for compliance frameworks that require evidence of access controls.

- **Supports Zero-Trust Architecture**: Secret management is a foundational component of zero-trust networking. Applications authenticate to the vault using workload identities (e.g., Kubernetes service accounts, AWS IAM roles), not static IP addresses or shared secrets. This aligns with the principle of never trusting implicit network location.

**Key Takeaways**

- Centralize all secrets in a hardened vault with dynamic, short-lived credentials to minimize exposure windows and eliminate hard-coded secrets from code and config.
- Enforce policy-based access control tied to workload identities, not network location, and ensure every secret access is logged for audit and forensics.
- Automate secret rotation and integrate secret retrieval into application startup and CI/CD pipelines to maintain security without manual overhead.
