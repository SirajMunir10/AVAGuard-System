================================================================================
SECTION 12: MOBILE APPLICATION PENETRATION TESTING
================================================================================

Mobile application penetration testing covers both Android and iOS platforms and requires specialized tools and techniques distinct from web application testing. The primary testing areas are static analysis of the application binary, dynamic analysis during runtime, data storage security, network traffic analysis, and platform-specific security feature implementation.

Android static analysis begins with obtaining the APK file either from the Google Play Store using tools like APKPure or directly from the device. The APK is decompiled using jadx to produce readable Java or Kotlin source code that can be reviewed for hardcoded secrets, insecure cryptographic implementations, exported components that can be called by other applications, and dangerous permission requests. The AndroidManifest.xml file is particularly important as it defines all activities, services, broadcast receivers, and content providers along with whether they are exported and what permissions they require. Exported components accessible without authentication represent a common attack vector.

Android dynamic analysis uses a rooted test device or emulator running Magisk to bypass root detection and enable Frida-based runtime instrumentation. Objection, built on Frida, automates common tasks such as bypassing SSL pinning, enumerating files written to the device, tracing method calls, and hooking authentication functions. All network traffic should be proxied through Burp Suite to capture and analyze API calls. Shared preferences files, SQLite databases written to the device, and logcat output should be examined for sensitive data leakage.

iOS static analysis requires extracting the application from a device as iOS apps are encrypted in the App Store distribution. Tools like frida-ios-dump can decrypt and extract the IPA from a jailbroken test device. The extracted application binary and supporting files can then be analyzed using tools like Ghidra or class-dump to recover the Objective-C class structure. The Info.plist file should be reviewed for App Transport Security exemptions that allow cleartext HTTP connections, dangerous URL scheme registrations, and embedded secrets. The app binary should be checked for disabled protections including PIE, stack canaries, and ARC.

SSL pinning is a common protection in mobile applications that prevents interception of API traffic by refusing connections to servers not presenting a specific certificate or public key. Testing SSL pinning bypass is essential for capturing API traffic during dynamic analysis. On Android, the network security configuration can sometimes be patched in the manifest, or Frida scripts can hook the certificate validation methods. On iOS, Frida and Objection's ios sslpinning disable command hooks common pinning implementations. Custom pinning implementations may require reverse engineering the binary to identify the specific validation function.

Data storage security testing examines all locations where the application writes data to the device and verifies that sensitive data is appropriately protected. On Android this includes shared preferences files at /data/data/package/shared_prefs, SQLite databases at /data/data/package/databases, files written to external storage, and content provider data. On iOS this includes the NSUserDefaults plist, the Keychain, SQLite databases in the Documents or Library directory, and clipboard data. Sensitive data such as authentication tokens, personally identifiable information, and health data should never be written to shared preferences, external storage, or unencrypted databases.
