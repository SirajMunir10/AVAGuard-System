Auditd, the Linux Audit Daemon, is the userspace component of the Linux Auditing System. Its primary function is to write audit records generated by the kernel to disk. Unlike system logging (syslog/rsyslog), which handles application-level messages, auditd captures security-relevant events at the kernel level, such as system calls, file accesses, and user authentication attempts. This makes it a mandatory tool for compliance frameworks like PCI DSS, HIPAA, and FedRAMP, which require detailed, tamper-proof logs of privileged actions and access to sensitive data.

**Key Components of Auditd Configuration**

The audit system is controlled through several configuration files and utilities. Understanding each is critical for effective deployment.

- **`/etc/audit/auditd.conf`**: The main daemon configuration file. It controls runtime behavior, including log file location (`log_file`), maximum log size (`max_log_file`), number of rotated logs (`num_logs`), and action when disk is full (`space_left_action`, `admin_space_left_action`). For compliance, set `max_log_file_action = ROTATE` and `space_left_action = EMAIL` or `SYSLOG`. Never leave `action_mail_acct` unset.

- **`/etc/audit/rules.d/audit.rules`**: The rules file that defines which events to capture. Rules are loaded at daemon start via `auditctl`. The file uses a specific syntax: `-a` (append rule), `-w` (watch a file), `-k` (key for filtering). Example: `-w /etc/passwd -p wa -k passwd_changes` watches for write and attribute changes to the password file.

- **`/etc/audit/plugins.d/`**: Directory for plugin configurations, such as `audisp-json` or custom scripts. Plugins can forward audit records to external SIEMs or syslog. For example, `active = yes` in `af_unix.conf` enables forwarding via a Unix socket.

- **`auditctl`**: Command-line utility to load, list, and manage rules at runtime. Use `auditctl -l` to list current rules. Rules added with `auditctl` are not persistent; they must be saved to `audit.rules` for permanence.

- **`aureport`**: Generates summary reports from audit logs. Useful for daily compliance checks. Example: `aureport -au` shows authentication attempts; `aureport -m` shows account modifications.

- **`ausearch`**: Queries audit logs with specific criteria. Example: `ausearch -k passwd_changes -ts today` retrieves all events tagged with the key `passwd_changes` from today.

**Real-World Application: Configuring Auditd for a Production Web Server**

Consider a production web server hosting a PCI DSS-compliant application. The goal is to monitor access to cardholder data, changes to critical configuration files, and all privileged command execution.

1.  **Set Up the Daemon Configuration (`auditd.conf`)**:
    - `log_file = /var/log/audit/audit.log`
    - `log_group = root`
    - `max_log_file = 100` (MB)
    - `num_logs = 10`
    - `max_log_file_action = ROTATE`
    - `space_left_action = EMAIL`
    - `admin_space_left_action = HALT` (prevents log loss, but test in staging)
    - `disk_full_action = HALT`
    - `action_mail_acct = root`

2.  **Define Audit Rules (`audit.rules`)**:
    - **File System Watches**:
      - `-w /etc/passwd -p wa -k identity`
      - `-w /etc/shadow -p wa -k identity`
      - `-w /etc/ssh/sshd_config -p wa -k ssh_config`
      - `-w /var/www/html/config.php -p wa -k web_config` (if config contains sensitive data)
    - **System Call Rules**:
      - `-a always,exit -F arch=b64 -S execve -F uid=0 -k root_command` (captures all commands run as root)
      - `-a always,exit -F arch=b64 -S open,openat -F dir=/var/www/html/private -F success=1 -k cardholder_data` (monitors reads to sensitive directory)
    - **Immutable Rule** (load last): `-e 2` (makes rules immutable until reboot, preventing tampering).

3.  **Enable and Test**:
    - Run `systemctl enable auditd && systemctl restart auditd`.
    - Verify rules: `auditctl -l`.
    - Generate a test event: `touch /etc/passwd` (as root) then `ausearch -k identity -ts recent`.

4.  **Forward to SIEM**:
    - Configure `audisp-json` plugin in `/etc/audit/plugins.d/au-json.conf` to send JSON-formatted logs to a remote syslog server. This ensures centralized monitoring and alerting.

**Security Relevance**

Auditd directly addresses several critical security and compliance requirements:

- **Accountability**: By logging every `execve` call by root, you create an undeniable record of who ran what command, even if they switch users via `su` or `sudo`. This is essential for incident response and forensic analysis.

- **Integrity Monitoring**: File watches on `/etc/passwd`, `/etc/shadow`, and `/etc/ssh/sshd_config` detect unauthorized modifications. Combined with `aureport`, you can generate daily reports of any changes to these files.

- **Tamper Resistance**: The immutable rule (`-e 2`) prevents an attacker from disabling auditing after gaining root access. Combined with log rotation and remote forwarding, logs survive host compromise.

- **Compliance Mapping**: PCI DSS Requirement 10.2 mandates logging of all access to cardholder data. A rule like `-w /var/www/html/private -p r -k cardholder_data` satisfies this. HIPAA requires audit controls for ePHI access; auditd provides the raw data.

- **Performance Considerations**: Overly broad rules (e.g., `-a always,exit -S all`) can degrade system performance. Always scope rules to specific directories, users, or system calls. Use `-F` filters to reduce noise. For high-traffic systems, consider using `-a exclude,always -F msgtype=...` to drop benign events like `CWD` (current working directory) records.

**Key Takeaways**
- Auditd captures kernel-level events essential for compliance and forensic analysis, distinct from application-level syslog.
- Effective configuration requires precise rules in `audit.rules`, proper daemon settings in `auditd.conf`, and immutable rules to prevent tampering.
- Always scope rules tightly to relevant files, users, and system calls to balance security visibility with system performance.
