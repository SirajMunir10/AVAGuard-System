================================================================================
SECTION 13: API SECURITY TESTING
================================================================================

API security testing has become one of the most important aspects of modern penetration testing as organizations increasingly expose functionality through REST APIs, GraphQL endpoints, and webhooks. The OWASP API Security Top 10 provides the primary taxonomy for API vulnerabilities and should be applied systematically to every API in scope.

Broken Object Level Authorization is the most prevalent and impactful API vulnerability. It occurs when an API endpoint accepts a user-supplied object identifier such as a user ID, account number, or document ID without verifying that the requesting user is authorized to access that specific object. Testing involves capturing legitimate API requests and systematically modifying all object identifiers to reference objects belonging to other users. Even a single endpoint vulnerable to BOLA can expose the entire user base's data through automated enumeration. API documentation, intercepted traffic, and source code review should be used to build a complete map of all object identifiers in use before beginning BOLA testing.

Broken Function Level Authorization occurs when an API exposes administrative or privileged functionality to regular users without proper authorization checks. Testing involves attempting to access endpoints that appear to be restricted to administrators or internal services, such as paths containing admin, internal, management, or v2. HTTP method manipulation, where a POST request is substituted with a PUT or DELETE, can sometimes bypass function-level authorization checks that only validate the HTTP method rather than the full action being performed. API documentation for private or partner tiers sometimes reveals administrative endpoints that can be tested from a standard user account.

Mass assignment vulnerabilities occur when an API binds user-supplied JSON or form data directly to an internal data model without filtering which fields are allowed to be modified. If an API endpoint allows updating a user profile and the underlying user object contains an isAdmin or role field, an attacker may be able to elevate privileges by including those fields in the update request. Testing involves identifying all object update endpoints and attempting to set fields that should be read-only or restricted, such as user roles, account balances, verification status, and subscription tier.

GraphQL APIs require specific testing techniques due to their flexible query structure. Introspection queries can be used to retrieve the complete API schema including all types, queries, mutations, and subscriptions, which provides a complete map of the attack surface even if no documentation is available. GraphQL APIs are susceptible to the same authorization vulnerabilities as REST APIs, but their flexible query structure also introduces risks including deeply nested queries that can cause denial of service, batched queries that bypass rate limiting, and subscription functionality that can enable persistent data access after authorization is revoked.
