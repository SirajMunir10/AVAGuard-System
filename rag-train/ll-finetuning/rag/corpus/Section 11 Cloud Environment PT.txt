================================================================================
SECTION 11: CLOUD ENVIRONMENT PENETRATION TESTING
================================================================================

Cloud penetration testing requires a fundamentally different approach from traditional network testing because the attack surface is defined by identity and access management configurations rather than network topology. The most impactful cloud vulnerabilities are rarely exploitable software bugs and are instead misconfigurations in IAM policies, storage access controls, and service configurations.

AWS IAM penetration testing begins with enumerating the permissions available to the compromised identity, whether that is a set of access keys discovered through OSINT, an EC2 instance metadata credential obtained through SSRF, or a user account obtained through phishing. The Pacu framework automates privilege enumeration and escalation attacks in AWS environments. Common privilege escalation paths include iam:PassRole combined with the ability to create Lambda functions or EC2 instances, iam:CreatePolicyVersion to create a new policy version with administrative permissions, and sts:AssumeRole to impersonate higher-privileged roles. Every enumerated IAM policy should be reviewed for overly permissive wildcards such as Action star on Resource star.

S3 bucket misconfigurations represent one of the most common and impactful cloud security findings. Publicly accessible buckets containing sensitive data, customer information, backups, or source code have been responsible for some of the largest data breaches in recent years. Testing involves attempting to list and read bucket contents both without credentials and with the compromised identity's credentials. Bucket policy and access control list configurations should be reviewed to identify any grants to the AllUsers or AuthenticatedUsers principals. The aws s3 ls command with the no-sign-request flag tests for unauthenticated access.

Azure Active Directory assessments focus on OAuth application permissions, conditional access policy gaps, and service principal configurations. The illicit consent grant attack abuses OAuth to trick an administrator into granting a malicious application access to the organization's Azure AD tenant. Password spray attacks against Azure AD must account for Smart Lockout by distributing attempts below the lockout threshold and across multiple IP addresses. ROADtools and AADInternals are the primary toolsets for Azure AD enumeration and attack simulation.

Kubernetes cluster penetration testing targets the API server, which is the central control plane component. An exposed Kubernetes API server accessible from the internet with weak authentication represents complete compromise of all containerized workloads. Testing should verify that anonymous authentication is disabled, that RBAC policies follow least privilege principles, that the service account token automounting is disabled for pods that do not need API access, and that pod security policies or admission controllers prevent the creation of privileged pods. A privileged pod with host namespace access can escape to the underlying node and compromise all other workloads on that node.

Cloud metadata endpoints are a critical attack target during web application testing against cloud-hosted applications. The AWS instance metadata service at 169.254.169.254 provides IAM role credentials to any process running on the EC2 instance. An SSRF vulnerability in a web application can be used to retrieve these credentials and pivot from the application to the cloud environment. Testers should always attempt to reach the metadata endpoint through any discovered SSRF and document the IAM permissions available through the instance role.
