AWS Key Management Service (KMS) is a managed service that enables you to create, control, and rotate encryption keys used to protect data across AWS services and within your applications. It integrates with AWS CloudTrail for auditing key usage and AWS Identity and Access Management (IAM) for fine-grained access control. KMS uses hardware security modules (HSMs) validated under FIPS 140-2 to protect key material.

**Key Components**

- **Customer Master Keys (CMKs):** The primary resource in KMS. A CMK is a logical representation of a master key that includes key material, metadata (key ID, creation date, description, key state), and a policy. CMKs can be symmetric (used for both encryption and decryption) or asymmetric (used for sign/verify or encrypt/decrypt with a public/private key pair). Symmetric CMKs are the default and most common.
- **Data Keys:** Encryption keys generated by KMS using a CMK. Data keys are not stored or managed by KMS; you retrieve them, use them locally to encrypt data, and then discard them. KMS provides plaintext and ciphertext versions of the data key. The ciphertext can be stored alongside the encrypted data for later decryption.
- **Key Policies:** The primary method for controlling access to a CMK. Each CMK has exactly one key policy, which is a JSON document that defines who can use the key and under what conditions. You can also use IAM policies in conjunction with key policies, but the key policy must explicitly grant the IAM policy permission to apply.
- **Grants:** An alternative to key policies for granting temporary, scoped access to a CMK. Grants are useful for cross-account access or for delegating permissions to AWS services (e.g., allowing S3 to use a CMK for SSE-KMS). Grants have a principal, operations, and optional conditions (e.g., encryption context).
- **Key Rotation:** Automatic rotation of the key material for a symmetric CMK. When enabled, KMS rotates the key material annually. The old key material is retained so that data encrypted with previous versions can still be decrypted. Manual rotation is also possible by creating a new CMK and updating aliases.
- **Key States:** CMKs have states that affect their availability: Enabled, Disabled, PendingDeletion, PendingImport, and Unavailable. Disabled keys cannot be used for cryptographic operations. PendingDeletion has a configurable waiting period (7-30 days) during which the key can be canceled.
- **Aliases:** Friendly names for CMKs that can be used in place of key IDs in API calls. Aliases are not tied to a specific key region and can be updated to point to a different CMK, enabling key rotation without changing application code.
- **Encryption Context:** A set of key-value pairs that provides additional authenticated data (AAD) during encryption and decryption. The same context must be provided during decryption. This binds the ciphertext to the context, preventing unauthorized decryption even if the key is compromised.

**Real-World Application**

A common pattern is envelope encryption, where a CMK encrypts a data key, and that data key encrypts the actual data. This approach balances performance and security: symmetric encryption of large data is fast, while the CMK protects the data key.

Example workflow for encrypting a file in an S3 bucket using SSE-KMS:
1. An application calls KMS GenerateDataKey with a specific CMK ID and an encryption context (e.g., `{"department":"finance"}`).
2. KMS returns a plaintext data key and a ciphertext blob containing the encrypted data key.
3. The application uses the plaintext data key to encrypt the file locally (e.g., using AES-256-GCM).
4. The application uploads the encrypted file and the ciphertext blob to S3. The encryption context is also stored as metadata.
5. The application securely discards the plaintext data key.
6. For decryption, the application downloads the ciphertext blob and calls KMS Decrypt with the same encryption context. KMS returns the plaintext data key.
7. The application uses the plaintext data key to decrypt the file.

For cross-account access, you create a grant on the CMK in the source account, specifying the principal in the target account. The target account can then use the CMK for specific operations (e.g., encrypt, decrypt) without needing a key policy update.

For compliance with key rotation requirements, enable automatic rotation on symmetric CMKs. For asymmetric CMKs, rotation is manual because the public key is distributed. You must create a new CMK and update references.

**Security Relevance**

KMS is foundational to data protection in AWS because it centralizes key management and enforces separation of duties. The key policy model ensures that no single entity can both manage the key and use it to decrypt data without explicit authorization. CloudTrail logs every KMS API call, providing an immutable audit trail for forensic analysis.

Common security pitfalls include:
- **Overly permissive key policies:** Granting `kms:*` to broad principals or not using conditions like `aws:SourceArn` to restrict usage to specific resources.
- **Missing encryption context:** Without a context, an attacker who gains access to a CMK can decrypt any ciphertext encrypted with that key. Always use a unique, verifiable context.
- **Not rotating keys:** Long-lived keys increase the blast radius if compromised. Enable automatic rotation for symmetric CMKs.
- **Using the same CMK for multiple environments:** Separate CMKs for development, staging, and production to limit the impact of a breach.
- **Storing plaintext data keys:** Data keys must be discarded immediately after use. Never persist them to disk or logs.

For high-security workloads, consider using AWS CloudHSM with KMS custom key stores. This allows you to generate and manage key material in a single-tenant HSM under your control, while still using KMS for key management and auditing.

**Key Takeaways**
- Always use an encryption context with every KMS encrypt and decrypt call to bind ciphertext to specific metadata and prevent unauthorized decryption.
- Enable automatic key rotation for all symmetric CMKs to limit the lifespan of key material and reduce the impact of a key exposure.
- Restrict key policies using conditions like `aws:SourceArn` and `kms:EncryptionContextKeys` to enforce least privilege and prevent cross-service misuse.
