**Definition**

AWS Secrets Manager and AWS Systems Manager Parameter Store are two managed services for securely storing, retrieving, and managing sensitive configuration data. Secrets Manager is purpose-built for secrets such as database credentials, API keys, and OAuth tokens, with built-in automatic rotation. Parameter Store is a broader, lower-cost service for storing configuration data, including plaintext strings, encrypted parameters, and hierarchical parameter paths. Both services integrate with AWS Key Management Service (KMS) for encryption at rest, but differ in feature depth, cost model, and use-case suitability.

**Key Components**

**Secrets Manager**
- **Secret storage**: Stores secrets as encrypted blobs (up to 64 KB) with versioning. Each secret has a unique ARN and can be tagged.
- **Automatic rotation**: Supports scheduled rotation via Lambda functions. AWS provides pre-built rotation templates for RDS, Redshift, DocumentDB, and other services. Custom rotation logic can be implemented.
- **Fine-grained access control**: IAM policies and resource-based policies control who can read, write, rotate, or delete secrets. Supports cross-account access.
- **Secret versioning**: Each secret has a staging label (e.g., AWSCURRENT, AWSPREVIOUS) to manage rotation transitions. Applications can pin to a specific version.
- **Integration**: Native integration with RDS, Redshift, DocumentDB, and other AWS services for automatic credential management. Also integrates with CloudFormation, ECS, and Lambda.
- **Audit trail**: All secret access and rotation events are logged in AWS CloudTrail.

**Parameter Store**
- **Parameter types**: String, StringList, and SecureString. SecureString parameters are encrypted with KMS. Plaintext parameters are stored as-is.
- **Tiered storage**: Standard tier (free, up to 10,000 parameters, 4 KB max) and Advanced tier (paid, up to 100,000 parameters, 8 KB max, with parameter policies).
- **Parameter policies**: Advanced tier supports expiration, expiration notification, and no-change notification policies. Useful for enforcing rotation or deprecation.
- **Hierarchical naming**: Supports path-based naming (e.g., /app/prod/db/password) for organizing parameters. IAM policies can grant access to entire paths.
- **Versioning**: Each parameter update increments a version number. Applications can reference a specific version or use the latest.
- **Integration**: Integrated with EC2 Systems Manager, CloudFormation, Lambda, ECS, and Step Functions. Can be referenced in CloudFormation templates using dynamic references.

**Real-World Application**

**Database credential rotation in a microservices architecture**
A company runs a multi-service application on ECS Fargate. Each service connects to an RDS PostgreSQL database using credentials stored in Secrets Manager. The security team configures automatic rotation every 30 days using the built-in RDS rotation template. The Lambda rotation function updates the secret and the database password simultaneously. Services retrieve the secret at startup and cache it for the session duration. If a rotation occurs mid-session, the service catches a connection error and re-fetches the secret, which now contains the new password. CloudTrail logs show all secret retrieval events, enabling audit trails for compliance.

**Configuration management across environments**
A DevOps team manages three environments (dev, staging, prod) for a web application. They use Parameter Store with hierarchical paths: /app/{environment}/db/url, /app/{environment}/api/key, /app/{environment}/feature/flags. The prod environment uses Advanced tier parameters with expiration policies to force quarterly review of API keys. IAM roles for each environment’s EC2 instances have policies that grant read access only to their respective path prefix. This prevents a dev instance from reading prod credentials. CloudFormation templates reference these parameters using dynamic references (resolve:ssm), ensuring infrastructure-as-code does not embed secrets.

**CI/CD pipeline secret injection**
A CI/CD pipeline (Jenkins or CodePipeline) needs to deploy an application that requires a third-party API key. The pipeline assumes an IAM role that has permission to read a specific secret in Secrets Manager. During deployment, the pipeline retrieves the secret and injects it as an environment variable into the container. The secret is never stored in the pipeline’s configuration files or logs. If the API key is compromised, the security team rotates it in Secrets Manager, and the next deployment automatically uses the new value.

**Security Relevance**

**Secrets Manager**
- **Reduces secret sprawl**: Centralizes secrets, eliminating hardcoded credentials in code, config files, or environment variables.
- **Enforces rotation**: Automatic rotation reduces the window of exposure if a secret is leaked. Without rotation, a compromised credential remains valid indefinitely.
- **Granular access control**: IAM policies can restrict secret access to specific roles, services, or even specific versions. This limits blast radius.
- **Audit trail**: Every secret access is logged. Security teams can detect anomalous retrieval patterns (e.g., a Lambda function reading a secret it never used before).
- **Encryption at rest and in transit**: Secrets are encrypted with KMS. Access to the KMS key is separate from access to the secret, adding a defense-in-depth layer.

**Parameter Store**
- **SecureString encryption**: Encrypts sensitive parameters (e.g., API keys, database passwords) with KMS. Plaintext parameters (e.g., URLs, feature flags) remain unencrypted but are still protected by IAM.
- **Path-based isolation**: Hierarchical naming allows fine-grained IAM policies. For example, a policy can grant read access to /app/prod/* but deny access to /app/dev/*. This prevents cross-environment data leaks.
- **Parameter policies**: Expiration policies force periodic review or rotation of sensitive parameters. No-change notification policies alert if a parameter hasn’t been updated, which may indicate a stale credential.
- **Cost-effective for non-secret data**: For configuration data that is not a secret (e.g., instance IDs, region names), Parameter Store avoids the cost of Secrets Manager while still providing access control and versioning.

**Key Takeaways**
1. Use Secrets Manager for secrets that require automatic rotation, version staging, and native integration with AWS database services; use Parameter Store for configuration data, especially when hierarchical organization and cost efficiency are priorities.
2. Always encrypt sensitive parameters using KMS (SecureString in Parameter Store, default in Secrets Manager) and restrict access via IAM policies that follow least privilege, including path-based restrictions for Parameter Store.
3. Enable CloudTrail logging for both services and monitor for unusual retrieval patterns (e.g., high frequency, unexpected source IPs) to detect potential credential compromise or misuse.
