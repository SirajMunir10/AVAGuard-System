Service accounts are non-human identities used by applications, services, scripts, and automated processes to authenticate and authorize against APIs, databases, and other resources. Unlike user accounts, service accounts are not tied to a single person and often have elevated privileges, long lifespans, and no requirement for interactive logon. Their security posture directly impacts the blast radius of a compromise, as a breached service account can grant an attacker persistent, privileged access to critical infrastructure.

**Key Components of Service Account Security**

Effective service account security requires control over four primary dimensions: lifecycle management, credential hygiene, privilege minimization, and monitoring.

- **Lifecycle Management:** Every service account must have a documented owner, a defined purpose, and an expiration date. Orphaned accounts—those created for a project that no longer exists—are a leading source of credential exposure. Implement a process to review and deprovision accounts quarterly. Use infrastructure-as-code (IaC) templates to enforce naming conventions and metadata tags (e.g., owner, application, environment) at creation time.

- **Credential Hygiene:** The most common failure is using static, long-lived secrets (passwords, API keys, or client secrets). These should be replaced with:
  - **Managed identities** (e.g., AWS IAM Roles for EC2, Azure Managed Identities, GCP Service Account Impersonation) that rotate credentials automatically.
  - **Short-lived tokens** (e.g., OAuth 2.0 tokens with a TTL of 1 hour or less) obtained via a secure token service.
  - **Secret rotation** enforced by a secrets manager (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault) with a maximum rotation interval of 90 days, and ideally 30 days for high-privilege accounts.
  - **No hardcoded secrets** in source code, configuration files, or environment variables. Use a secrets injection mechanism at runtime.

- **Privilege Minimization:** Apply the principle of least privilege strictly. A service account should only have the permissions required for its specific function, and no more. Key practices include:
  - **Role-based access control (RBAC)** with custom roles that grant only the necessary API actions (e.g., read-only on a specific S3 bucket, not full S3 access).
  - **Resource-level scoping** (e.g., allow access to a single database, not all databases in the instance).
  - **Deny-by-default policies** that explicitly block actions not required.
  - **Avoiding wildcard permissions** (`*`) in any policy statement.
  - **Using conditions** (e.g., IP address range, time of day, MFA session) to further restrict access.

- **Monitoring and Auditing:** Without visibility, you cannot detect misuse. Implement:
  - **Centralized logging** of all service account authentication events (success and failure) and authorization decisions (allow/deny).
  - **Anomaly detection** rules for unusual patterns: authentication from unexpected geographies, off-hours activity, or a sudden spike in API calls.
  - **Alerting on credential changes** (key rotation, password reset) and permission modifications.
  - **Regular access reviews** where owners must attest that each account’s permissions are still required.

**Real-World Application: Securing a CI/CD Pipeline**

Consider a typical CI/CD pipeline using Jenkins, GitHub Actions, or GitLab CI. The pipeline needs to deploy code to a production Kubernetes cluster and push container images to a private registry.

- **Insecure approach:** A single service account with a long-lived API token stored as a Jenkins credential. The token has cluster-admin privileges in Kubernetes and full push access to the registry. If the token is leaked (e.g., via a log file or a compromised developer workstation), an attacker can deploy malicious containers, exfiltrate images, or escalate to cluster-wide control.

- **Secure approach:**
  1. **Use workload identity federation:** Configure the CI system to assume an IAM role (e.g., AWS IAM Roles for Service Accounts, Azure Workload Identity) using OIDC tokens. No static secret is stored.
  2. **Short-lived credentials:** The pipeline requests a token with a 15-minute TTL, scoped to a specific Kubernetes namespace and a specific registry repository.
  3. **Least privilege:** The Kubernetes role only allows `create` and `patch` on `deployments` and `services` in the `production` namespace. The registry role only allows `push` to the `myapp` repository.
  4. **Audit trail:** All API calls are logged to CloudTrail and a SIEM. An alert triggers if the pipeline attempts to access a different namespace or registry.
  5. **Rotation:** The OIDC provider’s public keys are rotated automatically. No manual key management is needed.

This design eliminates the risk of a leaked static credential and limits the blast radius to a single namespace and repository.

**Security Relevance**

Service accounts are a prime target for attackers because they often bypass security controls designed for human users (e.g., MFA, session timeouts, behavioral analytics). A compromised service account can be used to:

- **Lateral movement:** Access databases, storage buckets, and other services without triggering user-based alerts.
- **Privilege escalation:** If the account has permission to modify IAM policies or create new accounts, an attacker can grant themselves full administrative access.
- **Data exfiltration:** Read sensitive data from databases, object stores, or APIs.
- **Persistence:** Create backdoor accounts or modify infrastructure to maintain access after the initial breach is remediated.

The 2023 Microsoft Digital Defense Report noted that 70% of identity-related breaches involved non-human identities, underscoring that service accounts are a weak link in most IAM implementations. Treating them with the same rigor as privileged user accounts—or greater, given their lack of human oversight—is essential for a mature security posture.

**Key Takeaways**
- Replace static, long-lived credentials with managed identities or short-lived tokens obtained via a secrets manager or workload federation.
- Apply least privilege at the resource and action level, using custom roles and conditions, and never use wildcard permissions.
- Implement continuous monitoring and regular access reviews to detect and remediate orphaned or over-privileged service accounts.
