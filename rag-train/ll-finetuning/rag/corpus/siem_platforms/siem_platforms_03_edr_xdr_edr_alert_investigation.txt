**Definition**

EDR alert investigation is the systematic process of analyzing endpoint detection and response (EDR) telemetry to determine whether a triggered alert represents a genuine security incident, a false positive, or an environmental anomaly. Unlike simple signature-based alerts, EDR alerts are generated by behavioral detection engines, machine learning models, and threat intelligence correlations. The investigation aims to reconstruct the full attack chain—from initial access to impact—by examining process trees, network connections, file system modifications, registry changes, and memory artifacts. The output of an investigation is a validated verdict (malicious, benign, or suspicious) and, if malicious, a containment and remediation plan.

**Key Components of an EDR Alert Investigation**

**1. Alert Triage and Contextualization**
- Review the alert metadata: detection rule name, severity, MITRE ATT&CK technique mapping, and the specific endpoint(s) involved.
- Check the endpoint’s asset criticality, business unit, and current user context. A PowerShell execution on a domain controller warrants higher priority than on a developer workstation.
- Correlate the alert timestamp with known maintenance windows, software deployments, or user activity logs to rule out scheduled tasks.

**2. Process Tree Analysis**
- Examine the parent-child process chain. For example, `winword.exe` spawning `cmd.exe` which then launches `powershell.exe` is a classic indicator of macro-based malware.
- Identify anomalous process relationships: `svchost.exe` executing `rundll32.exe` with no DLL argument, or `explorer.exe` spawning `wscript.exe`.
- Check process command-line arguments for obfuscation patterns (base64 encoding, use of `-enc`, `-e`, or `-Command` with long strings).

**3. Network Connection Mapping**
- List all outbound connections initiated by the suspicious process within the alert timeframe.
- Compare destination IPs and domains against threat intelligence feeds, known C2 infrastructure, and internal allowlists.
- Look for beaconing patterns: regular intervals (e.g., every 60 seconds), small packet sizes, or connections to non-standard ports (e.g., 4444, 8080, 8443).

**4. File and Registry Artifact Examination**
- Identify files created, modified, or deleted by the process. Focus on dropped executables, DLLs, scripts, and persistence mechanisms.
- Check registry run keys (`HKLM\Software\Microsoft\Windows\CurrentVersion\Run`), scheduled tasks, and service creation.
- Hash any suspicious files and query VirusTotal or internal malware analysis sandbox for reputation.

**5. Memory and Behavioral Indicators**
- If the EDR supports memory scanning, check for injected code, reflective DLL loading, or process hollowing.
- Look for unusual API calls: `VirtualAllocEx`, `WriteProcessMemory`, `CreateRemoteThread` in a non-debugger context.
- Verify whether the process attempted to disable security controls (e.g., stopping the EDR agent, modifying firewall rules).

**Real-World Application: Investigating a Lateral Movement Alert**

**Scenario:** An EDR alert fires for `wmic.exe` executing on a workstation with the command:
`wmic.exe /node:192.168.1.50 process call create "powershell.exe -enc <base64>"`

**Investigation Steps:**

1. **Triage:** The alert maps to MITRE ATT&CK T1047 (Windows Management Instrumentation). The source endpoint is a standard user workstation; the target IP is a file server.
2. **Process Tree:** On the source endpoint, `explorer.exe` spawned `cmd.exe`, which spawned `wmic.exe`. No legitimate administrative activity was scheduled.
3. **Network Connections:** The source endpoint had an SMB connection to the target IP (port 445) immediately before the alert. The target IP also showed inbound RDP connections from the same source in the preceding hour.
4. **Artifacts:** On the target file server, a scheduled task was created under `\Microsoft\Windows\TaskScheduler` with a base64-encoded PowerShell payload. Decoding the payload revealed a reverse shell to an external IP.
5. **Memory:** The target server’s `svchost.exe` showed signs of process injection via `CreateRemoteThread` targeting `lsass.exe`.

**Outcome:** The investigation confirmed active lateral movement using WMI, followed by credential dumping. The source workstation was isolated, the file server was quarantined, and the external C2 IP was blocked at the perimeter firewall.

**Security Relevance**

EDR alert investigation is the critical bridge between detection and response. Without thorough investigation, organizations face three primary risks:

- **Alert fatigue:** Uninvestigated false positives desensitize analysts, leading to missed true positives.
- **Incomplete containment:** A shallow investigation may stop one indicator (e.g., kill a process) while leaving persistence mechanisms or lateral movement paths active.
- **Forensic gaps:** Failure to collect and preserve artifacts during investigation can hinder post-incident analysis, legal proceedings, and regulatory compliance (e.g., GDPR breach notification requirements).

Effective investigation directly supports compliance frameworks such as NIST SP 800-61 (Computer Security Incident Handling), which mandates that organizations "analyze all relevant information to determine the scope of the incident." It also enables accurate reporting under PCI DSS Requirement 12.10.1, which requires timely detection and response to security events.

Furthermore, investigation findings feed back into detection engineering. Each confirmed incident provides ground truth data to tune detection rules, reduce false positives, and identify gaps in coverage. For example, if an investigation reveals that a PowerShell download cradle evaded detection, the SOC can create a new rule targeting that specific pattern.

**Key Takeaways**
- EDR alert investigation must reconstruct the full attack chain using process trees, network logs, file artifacts, and memory indicators to distinguish true incidents from benign anomalies.
- Real-world investigations require correlating multiple data sources (endpoint, network, identity) and following the evidence across systems to detect lateral movement and persistence.
- Thorough investigation reduces alert fatigue, ensures complete containment, and provides actionable feedback to improve detection rules and compliance posture.
