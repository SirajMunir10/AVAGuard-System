Kerberos is a network authentication protocol designed to provide strong authentication for client-server applications using secret-key cryptography. Within Active Directory, Kerberos is the default authentication protocol, replacing the older NTLM. It operates on the principle of issuing tickets that grant access to services without transmitting passwords over the network. Understanding its mechanics is critical for identifying attack vectors and hardening domain security.

**Key Components**

The Kerberos protocol relies on several distinct components, each with a specific role in the authentication flow:

- **Key Distribution Center (KDC):** A service that runs on every domain controller. It consists of two logical parts: the Authentication Service (AS) and the Ticket-Granting Service (TGS). The KDC holds a copy of every user and machine account password hash (derived from the NTLM hash or AES keys).
- **Authentication Service (AS):** Handles the initial authentication request. When a user logs in, the client sends an AS-REQ containing the user’s principal name. The AS responds with an AS-REP containing a Ticket-Granting Ticket (TGT) encrypted with the user’s password hash.
- **Ticket-Granting Service (TGS):** Issues service tickets. After obtaining a TGT, the client sends a TGS-REQ to request access to a specific service (e.g., a file server). The TGS validates the TGT and returns a service ticket encrypted with the target service’s password hash.
- **Ticket-Granting Ticket (TGT):** A temporary credential that proves the user has been authenticated by the AS. It contains the user’s Security Identifier (SID), group memberships, and a session key. The TGT is encrypted with the KDC’s long-term key (the `krbtgt` account hash).
- **Service Ticket (ST):** A ticket issued by the TGS that allows the client to authenticate to a specific service. It is encrypted with the service account’s password hash and includes a session key for the client-service communication.
- **Authenticator:** A timestamp and other data encrypted with the session key. The client sends this along with the service ticket to prove possession of the session key. The service decrypts the authenticator to verify the client’s identity.
- **Session Keys:** Temporary symmetric keys generated by the KDC for secure communication between the client and the KDC (TGT session key) and between the client and the target service (service session key).
- **Pre-Authentication:** A required step where the client proves knowledge of the user’s password hash before the AS issues a TGT. This prevents offline brute-force attacks against the AS-REP. The default method uses a timestamp encrypted with the user’s long-term key.

**Real-World Application**

In a typical Active Directory environment, Kerberos authentication proceeds as follows:

1.  **User Logon:** The user enters their password. The client derives a hash from the password.
2.  **AS Exchange:** The client sends an AS-REQ to the KDC (domain controller). The KDC looks up the user’s account, generates a TGT and a TGT session key, encrypts the TGT with the `krbtgt` account hash, and encrypts the TGT session key with the user’s password hash. The client decrypts the session key using the derived hash.
3.  **TGS Exchange:** The client sends a TGS-REQ containing the TGT and an authenticator (encrypted with the TGT session key) to request a service ticket for a specific service (e.g., CIFS/fileserver). The KDC decrypts the TGT using the `krbtgt` hash, validates the authenticator, checks the user’s group memberships, and issues a service ticket encrypted with the target service account’s password hash.
4.  **Service Authentication:** The client sends the service ticket and a new authenticator (encrypted with the service session key) to the target service. The service decrypts the ticket using its own password hash, extracts the session key, and decrypts the authenticator. If the timestamps are within tolerance (default 5 minutes), the service grants access.

This process is transparent to the user and occurs every time a user accesses a network resource. Service accounts, computer accounts, and user accounts all follow the same flow.

**Security Relevance**

Kerberos is generally more secure than NTLM because it does not transmit password hashes over the network and uses mutual authentication. However, several security considerations and attack vectors exist:

- **Golden Ticket Attack:** If an attacker compromises the `krbtgt` account hash (e.g., via DCSync), they can forge TGTs for any user, including domain admins. This grants persistent, undetectable access. Mitigation: Rotate the `krbtgt` password twice after a compromise, restrict DCSync permissions, and monitor for anomalous TGT requests.
- **Silver Ticket Attack:** An attacker who compromises a service account’s password hash can forge service tickets for that specific service. This allows access to the service without interacting with the KDC. Mitigation: Use managed service accounts (gMSA) with automatic password rotation, limit service account privileges, and enable Kerberos armoring (FAST).
- **Kerberoasting:** An attacker requests a service ticket for a service account with a user principal name (SPN). The service ticket is encrypted with the service account’s password hash. The attacker can then brute-force the hash offline. Mitigation: Use strong, complex passwords for service accounts, use gMSAs, and monitor for unusual TGS-REQ patterns.
- **AS-REP Roasting:** If a user account does not require Kerberos pre-authentication (a misconfiguration), an attacker can request a TGT for that user and receive an AS-REP encrypted with the user’s password hash, which can be brute-forced offline. Mitigation: Ensure all user accounts have pre-authentication enabled. Audit for accounts with `DONT_REQUIRE_PREAUTH` set.
- **Pass-the-Ticket (PtT):** An attacker extracts a TGT or service ticket from a compromised host and uses it to authenticate to other services. Mitigation: Use Credential Guard, restrict local administrator access, and implement network segmentation.
- **Time Synchronization:** Kerberos relies on synchronized clocks (default tolerance of 5 minutes). An attacker with the ability to manipulate system time could replay tickets. Mitigation: Use NTP with authentication, restrict time synchronization permissions.
- **Delegation:** Unconstrained delegation allows a service to impersonate a user to any other service. If the service is compromised, the attacker can impersonate any user connecting to it. Mitigation: Use constrained delegation or resource-based constrained delegation instead. Disable unconstrained delegation where possible.

**Key Takeaways**
- Kerberos is the default authentication protocol in Active Directory and relies on tickets and session keys, never transmitting passwords over the network.
- The most critical security risk is compromise of the `krbtgt` account hash, which enables Golden Ticket attacks; protect it with strict access controls and regular rotation.
- Common attack vectors like Kerberoasting, AS-REP Roasting, and Silver Tickets can be mitigated by enforcing strong service account passwords, enabling pre-authentication, and using managed service accounts.
