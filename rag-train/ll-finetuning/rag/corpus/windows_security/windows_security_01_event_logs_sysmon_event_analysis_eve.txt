Sysmon Event ID 1 (Process Creation) is a critical telemetry source for detecting malicious activity on Windows endpoints. It logs every process creation attempt, providing granular details that native Windows Event ID 4688 (Security Auditing) often lacks. For a security engineer, mastering Event ID 1 analysis is essential for building detection rules, conducting threat hunting, and performing incident response.

**Definition**

Event ID 1 is generated by the Sysinternals System Monitor (Sysmon) driver each time a new process is created on the system. Unlike the default Windows security event log, Sysmon enriches process creation data with fields such as the process GUID, parent process GUID, image path, command line, hashes (MD5, SHA1, SHA256, IMPHASH), and digital signature information. This event is the foundation for understanding process ancestry and execution flow on a host. It is enabled by default when Sysmon is installed with a configuration that includes the `ProcessCreate` rule.

**Key Components of Event ID 1**

Each Event ID 1 record contains the following fields, all of which are directly usable for analysis:

- **UtcTime**: Timestamp in UTC. Critical for correlating events across multiple systems and logs.
- **ProcessGuid**: A unique identifier for the process instance. This is the primary key for linking child processes (Event ID 1) and process termination (Event ID 5).
- **ProcessId**: The Windows process ID (PID). Note that PIDs are reused; use ProcessGuid for persistent tracking.
- **Image**: The full path to the executable file (e.g., `C:\Windows\System32\cmd.exe`).
- **FileVersion**: The version resource string from the executable. Useful for identifying known legitimate software versions.
- **Description**: The file description from the executable’s version info.
- **Product**: The product name from the version info.
- **Company**: The company name from the version info.
- **OriginalFileName**: The original name of the file as compiled. This is often more reliable than the Image name for identifying the true binary (e.g., `cmd.exe` vs. a renamed copy).
- **CommandLine**: The full command line used to launch the process. This is the most valuable field for detecting malicious arguments, encoded payloads, or LOLBins (Living Off the Land Binaries).
- **CurrentDirectory**: The working directory of the process. Can indicate execution from a temp folder or user download location.
- **User**: The account context under which the process runs (e.g., `NT AUTHORITY\SYSTEM` or `DOMAIN\user`).
- **LogonGuid**: Links the process to a specific logon session (Event ID 3). Essential for tracking lateral movement.
- **LogonId**: The logon session ID.
- **TerminalSessionId**: The session ID for interactive or remote desktop sessions.
- **IntegrityLevel**: The integrity level (e.g., High, Medium, Low, System). High or System integrity indicates elevated privileges.
- **Hashes**: Full hash set (MD5, SHA1, SHA256, IMPHASH). IMPHASH is particularly useful for clustering malware variants.
- **ParentProcessGuid**: The ProcessGuid of the parent process. This enables reconstruction of the process tree.
- **ParentProcessId**: The PID of the parent process.
- **ParentImage**: The full path of the parent executable.
- **ParentCommandLine**: The command line of the parent process. Critical for detecting parent-child anomalies (e.g., `winword.exe` spawning `powershell.exe`).
- **ParentUser**: The user context of the parent process.

**Real-World Application**

Event ID 1 is the primary data source for process-based detection rules. Here are specific, actionable use cases:

- **Detecting LOLBin Abuse**: Write a rule that flags when `rundll32.exe` or `regsvr32.exe` is spawned by a Microsoft Office application (e.g., `winword.exe`, `excel.exe`). This is a common technique for initial access via macro-enabled documents.
- **Command Line Anomalies**: Monitor for `cmd.exe` or `powershell.exe` executions with command lines containing base64-encoded strings, `-EncodedCommand`, or `-WindowStyle Hidden`. These are strong indicators of script-based attacks.
- **Parent-Child Mismatches**: Flag any process where `ParentImage` is a web browser (e.g., `chrome.exe`, `firefox.exe`) and the child process is a scripting engine (e.g., `wscript.exe`, `cscript.exe`, `powershell.exe`). This indicates a drive-by download or social engineering attack.
- **Unusual Image Paths**: Alert on any process executed from `%TEMP%`, `%APPDATA%`, `%LOCALAPPDATA%`, or `C:\Users\Public\` that is not a known installer. Malware frequently executes from these locations.
- **Hash-Based Blocking**: Use the `Hashes` field to block known malware hashes at the detection layer. IMPHASH can be used to block entire malware families with a single signature.
- **Process Hollowing Detection**: While Event ID 1 alone cannot detect hollowing, you can correlate it with Event ID 7 (Image Loaded) or Event ID 10 (Process Access) to identify a process that loads unexpected DLLs or has a mismatched original filename.

**Security Relevance**

Event ID 1 is foundational for the following security objectives:

- **Initial Access Detection**: The first process creation event from a phishing email or malicious website is often the earliest telemetry point. Analyzing the parent-child relationship and command line can reveal the attack vector.
- **Lateral Movement Tracking**: When a remote service (e.g., `svchost.exe` or `services.exe`) spawns `cmd.exe` or `powershell.exe` with a non-standard user context, it may indicate lateral movement via PsExec, WMI, or scheduled tasks.
- **Privilege Escalation**: A process running as a standard user that spawns a child process with `IntegrityLevel` of High or System is a strong indicator of privilege escalation (e.g., UAC bypass or exploit).
- **Persistence Monitoring**: Processes launched from startup folders, Run keys, or scheduled tasks will appear as Event ID 1 events. Correlating these with the parent process (e.g., `explorer.exe` for startup folder) helps identify persistence mechanisms.
- **Incident Response Timeline**: The `UtcTime` field provides a precise timeline of execution. During an incident, reconstructing the process tree from Event ID 1 records allows you to determine the exact sequence of commands executed by an attacker.

**Key Takeaways**
- Event ID 1 provides the most detailed process creation telemetry available on Windows, including parent GUID, command line, hashes, and digital signature data, which native logs do not offer.
- The most effective detection rules focus on anomalous parent-child relationships (e.g., Office spawning scripting engines) and suspicious command line arguments (e.g., base64 encoding, hidden windows).
- Always use `ProcessGuid` and `ParentProcessGuid` for process tree reconstruction, not PIDs, because PIDs are reused and can lead to incorrect parent attribution.
