Kerberos is a network authentication protocol designed to provide strong authentication for client-server applications using secret-key cryptography. Within Active Directory, Kerberos is the default authentication protocol, replacing the older NTLM. It operates on the principle of a trusted third-party model, where a central Key Distribution Center (KDC) issues tickets that grant access to network resources without transmitting passwords in plaintext.

**Key Components**

The Kerberos protocol relies on several distinct components that work in sequence to authenticate a user and authorize access to a service.

- **Key Distribution Center (KDC):** The core service that runs on every Domain Controller in Active Directory. It has two logical parts: the Authentication Service (AS) and the Ticket-Granting Service (TGS). The KDC maintains a database of all principal (user, computer, service) credentials, specifically their long-term secret keys derived from passwords or machine account secrets.
- **Authentication Service (AS):** The first point of contact for a client. When a user logs on, the client sends an AS-REQ (Authentication Service Request) containing the user's principal name. The AS responds with an AS-REP containing two items: a Ticket-Granting Ticket (TGT) encrypted with the KDC's own secret key, and a session key encrypted with the user's long-term key (derived from their password). The client decrypts the session key using its password hash.
- **Ticket-Granting Service (TGS):** Used when the client needs to access a specific service (e.g., a file server, web application). The client sends a TGS-REQ containing the TGT, an authenticator (encrypted with the TGT session key), and the Service Principal Name (SPN) of the target resource. The TGS validates the TGT, then issues a TGS-REP containing a service ticket (encrypted with the target service's long-term key) and a new session key for the client and service.
- **Ticket-Granting Ticket (TGT):** A temporary credential issued by the AS. It is encrypted with the KDC's master key (krbtgt account hash) and contains the user's identity, group memberships, and a session key. The TGT is used to request service tickets without re-entering credentials. Its lifetime is typically 10 hours by default in Windows domains.
- **Service Ticket:** A ticket issued by the TGS for a specific service. It is encrypted with the target service's long-term key (the machine account or service account password hash). The service ticket contains the user's identity and a session key for the client-service session.
- **Authenticator:** A timestamped, encrypted message sent by the client to prove possession of the session key. It prevents replay attacks because the receiving party checks the timestamp against a configurable skew (default 5 minutes in Windows).
- **Session Keys:** Temporary symmetric keys generated by the KDC for use between the client and the KDC (TGT session key) and between the client and the target service (service session key). These keys are never transmitted in plaintext.
- **Service Principal Name (SPN):** A unique identifier for a service instance. In Active Directory, an SPN is registered on a computer or service account object. The KDC uses the SPN to locate the correct account and its associated long-term key for encrypting the service ticket.

**Real-World Application**

In a standard Active Directory environment, the Kerberos authentication flow for a user accessing a file share proceeds as follows:

1.  **Initial Authentication (AS Exchange):** User 'jsmith' logs into a domain-joined workstation. The workstation sends an AS-REQ to the Domain Controller (KDC). The KDC checks if the user account exists and is not locked out or disabled. If valid, it creates a TGT and a session key. The TGT is encrypted with the krbtgt account's hash. The session key is encrypted with jsmith's password hash. The workstation uses jsmith's password hash (entered at logon) to decrypt the session key. The TGT is stored in the workstation's Kerberos ticket cache (LSASS process memory).
2.  **Service Request (TGS Exchange):** User jsmith double-clicks a mapped drive to `\\fileserver\share`. The workstation constructs a TGS-REQ. It includes the cached TGT, an authenticator encrypted with the TGT session key, and the SPN for the file server (e.g., `cifs/fileserver.contoso.com`). The KDC receives the request, decrypts the TGT using its own krbtgt hash, extracts the TGT session key, and uses it to decrypt the authenticator. It verifies the timestamp is within the allowed skew. If valid, the KDC looks up the SPN, finds the computer account for 'fileserver', and creates a service ticket encrypted with that computer account's password hash. It also generates a new session key for the client and service. The service ticket and new session key are sent back to the workstation.
3.  **Service Authentication (AP Exchange):** The workstation sends the service ticket and a new authenticator (encrypted with the service session key) to the file server. The file server receives the service ticket, decrypts it using its own machine account password hash, extracts the service session key, and uses it to decrypt the authenticator. It verifies the timestamp. Upon success, the file server reads the user's identity and group memberships from the service ticket. It then checks the file share's Access Control List (ACL) to determine if jsmith has the required permissions. Access is granted or denied based on the ACL.

**Security Relevance**

Kerberos is significantly more secure than NTLM, but it is not immune to attack. A security engineer must understand the following attack vectors and mitigations.

- **Golden Ticket Attack:** An attacker who gains access to the krbtgt account's password hash (typically via Domain Admin privileges and a DCSync attack) can forge TGTs. This allows the attacker to impersonate any user, including domain administrators, for the lifetime of the forged ticket (up to 20 years if not constrained). **Mitigation:** Regularly rotate the krbtgt password (twice, with a waiting period between resets) using a scheduled process. Implement privileged access workstations (PAWs) to protect Domain Admin credentials. Monitor for anomalous TGT requests, especially those with unusually long lifetimes.
- **Silver Ticket Attack:** An attacker who compromises a service account's password hash (e.g., a SQL Server service account or a machine account) can forge service tickets for that specific service. This allows access to the service without needing a TGT. **Mitigation:** Use Group Managed Service Accounts (gMSAs) which have automatic password rotation. Monitor for service ticket requests that do not have a corresponding TGS exchange (i.e., service tickets presented directly to a service without a prior KDC request). Enable Kerberos armoring (FAST) to protect the initial AS exchange.
- **Kerberoasting:** An attacker requests a service ticket for a service account (any account with an SPN). The service ticket is encrypted with the service account's password hash. The attacker can then extract the ticket from memory and perform offline brute-force or dictionary attacks against the hash. **Mitigation:** Ensure service accounts have strong, complex passwords (25+ characters). Use managed service accounts where possible. Monitor for an unusually high volume of TGS-REQ requests for a single SPN from a single source. Consider disabling RC4 encryption for Kerberos, forcing AES encryption which is computationally harder to crack.
- **AS-REP Roasting:** An attacker can request an AS-REP for a user account that does not require Kerberos pre-authentication. The AS-REP contains data encrypted with the user's password hash, which can be cracked offline. **Mitigation:** Ensure all user accounts have "Do not require Kerberos preauthentication" disabled. Audit for accounts with this setting enabled.
- **Pass-the-Ticket (PtT):** An attacker can extract a TGT or service ticket from a compromised workstation's LSASS memory and use it on another machine to impersonate the user. **Mitigation:** Implement Credential Guard on Windows 10/11 and Windows Server 2016+ to protect LSASS secrets. Use Protected Users Security Group to restrict NTLM and enforce AES encryption. Limit local administrator privileges on workstations.
- **Skeleton Key Attack:** A rootkit injected into a Domain Controller's LSASS process can allow any user to authenticate with a single, hardcoded password. **Mitigation:** Implement LSA Protection (RunAsPPL) on all Domain Controllers. Use Windows Defender Application Control (WDAC) or AppLocker to block unauthorized binaries. Monitor for changes to LSASS process memory.

**Key Takeaways**
- Kerberos relies on a trusted KDC and symmetric key cryptography; the security of the entire domain hinges on protecting the krbtgt account hash and all service account password hashes.
- The most critical attacks against Kerberos in Active Directory (Golden Ticket, Silver Ticket, Kerberoasting) all involve compromising or cracking secret key material, not breaking the protocol itself.
- Mitigations must focus on credential hygiene (strong passwords, managed accounts, regular krbtgt rotation), endpoint protection (Credential Guard, LSA Protection), and monitoring for anomalous ticket requests and lifetimes.
