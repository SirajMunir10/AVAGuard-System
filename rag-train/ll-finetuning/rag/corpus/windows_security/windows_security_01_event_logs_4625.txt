Event ID 4625 is a Windows Security log event that records a failed account logon attempt. It is generated by the Security Account Manager (SAM) on the target computer, regardless of whether the logon is local or remote. For domain-joined systems, a 4625 event is generated on the domain controller when authentication fails against the domain account database. This event is the direct counterpart to successful logon event 4624 and is a primary data source for detecting brute-force attacks, credential stuffing, and account enumeration.

**Key Components of Event ID 4625**

Every 4625 event contains a structured set of fields that provide the forensic details of the failure. The most critical fields for analysis are:

- **Account Name**: The username that was submitted during the failed logon attempt. This is the target account.
- **Account Domain**: The domain or machine name associated with the account. For local accounts, this is the computer name.
- **Logon Type**: An integer indicating the method of logon. Common values include:
    - 2 (Interactive) – logon at the console or via Remote Desktop (RDP) with saved credentials.
    - 3 (Network) – network logon, such as SMB, IIS, or SQL Server authentication.
    - 7 (Unlock) – workstation unlock after screen saver.
    - 8 (NetworkCleartext) – network logon where credentials are sent in plaintext, often seen with basic authentication.
    - 10 (RemoteInteractive) – Remote Desktop logon.
- **Failure Reason**: A text string describing why the logon failed. Common reasons include:
    - `%%2304`: An error occurred during logon (generic).
    - `%%2305`: The user has not been granted the requested logon type at this computer.
    - `%%2307`: The account is disabled.
    - `%%2313`: The specified account password has expired.
    - `%%2328`: The account is locked out.
- **Status**: A hexadecimal error code from the authentication package. The most common is `0xC000006D` (bad username or password). `0xC0000064` indicates the username does not exist.
- **Sub Status**: A more granular error code. For example, `0xC000006A` means the correct password was provided but the account is disabled. `0xC0000072` indicates the account is locked out.
- **Workstation Name**: The source machine name from which the logon attempt originated. This field is often blank for network logons.
- **Source Network Address**: The IP address of the remote machine that initiated the logon attempt. This is critical for identifying the attacker’s origin.
- **Process Name**: The executable path of the process that initiated the logon. This field is often empty for network logons but populated for interactive logons.
- **Logon ID**: A unique identifier for this logon session, which can be correlated with other events (e.g., 4634 for logoff).

**Real-World Application**

A security engineer uses 4625 events to build detection logic for authentication anomalies. The most common use case is detecting password spraying or brute-force attacks.

**Example Scenario: Password Spray Detection**

An attacker enumerates valid usernames via a web application and then attempts a single common password against each account. On the domain controller, this generates a burst of 4625 events with:
- Logon Type 3 (network)
- Status `0xC000006D` (bad password)
- Sub Status `0xC000006A` (correct password, but account disabled) if the account is disabled, or `0xC000006D` if the password is wrong.
- Source Network Address originating from a single IP or a small range.

A detection rule would aggregate 4625 events per Source Network Address over a short time window (e.g., 5 minutes) and trigger an alert if the count exceeds a threshold (e.g., 10 unique accounts). The rule must exclude known service accounts and legitimate authentication sources.

**Example Scenario: Account Lockout Investigation**

When a user reports being locked out, the engineer queries the domain controller’s security log for 4625 events with Sub Status `0xC0000072` (account locked out) for the affected account. The Source Network Address field reveals the attacking machine. If the source is a compromised workstation, the engineer can isolate it and check for malware. If the source is an external IP, the engineer blocks it at the firewall and reviews the authentication path.

**Example Scenario: Brute-Force Against RDP**

An external attacker attempts to brute-force RDP credentials. The target server generates 4625 events with Logon Type 10 (RemoteInteractive) and Status `0xC000006D`. The Source Network Address is the attacker’s IP. A detection rule that counts 4625 events with Logon Type 10 from a single IP within a short window can trigger an alert and automatically add a firewall block rule.

**Security Relevance**

Event ID 4625 is a cornerstone of identity threat detection. Its primary security value lies in:

- **Brute-Force Detection**: High volumes of 4625 events from a single source indicate an automated password guessing attack. Tuning thresholds per logon type prevents false positives from legitimate retry mechanisms.
- **Credential Stuffing**: A low volume of 4625 events across many accounts from a single source suggests credential stuffing, where the attacker has a list of compromised credentials and tests them against your environment.
- **Account Enumeration**: A 4625 event with Status `0xC0000064` (user does not exist) indicates the attacker is probing for valid usernames. This is a precursor to targeted attacks.
- **Insider Threat**: A user who repeatedly fails authentication against a sensitive server (e.g., a domain controller) may be attempting unauthorized access. Correlating 4625 events with the user’s normal behavior can reveal malicious intent.
- **Compliance**: Many regulatory frameworks (PCI DSS, SOX, HIPAA) require monitoring of failed authentication attempts. A SIEM query for 4625 events is the standard method for generating compliance reports.

**Limitations and Considerations**

- 4625 events are generated on the target machine, not the source. For network logons, the source IP may be a relay (e.g., a web server) rather than the attacker’s actual IP.
- The Workstation Name field is unreliable for network logons; it may be blank or contain the source machine’s NetBIOS name, which can be spoofed.
- Logon Type 3 events from legitimate services (e.g., scheduled tasks, service accounts) can generate high volumes of 4625 events if a service account password is changed without updating the service. These must be baselined and excluded from detection rules.
- Event ID 4625 does not capture the full authentication chain. For example, a failed Kerberos ticket request generates event ID 4768 (Kerberos Authentication Ticket Request) with a failure code, not a 4625. Both events must be monitored for complete coverage.

**Key Takeaways**
- Event ID 4625 is the definitive record of a failed logon attempt, containing the target account, source IP, logon type, and failure reason, which are essential for forensic analysis.
- Detection rules for brute-force and credential stuffing must aggregate 4625 events by source IP and logon type, with thresholds tuned to exclude legitimate service account behavior.
- Always correlate 4625 events with other security events (e.g., 4768, 4776, 4624) to distinguish between a password spray, a credential stuffing attack, and a simple misconfiguration.
