**Definition**

Credential Guard is a virtualization-based security (VBS) feature in Windows 10 Enterprise, Windows 11 Enterprise, and Windows Server 2016 and later that isolates domain credentials, NTLM hashes, and Kerberos tickets from the operating system. It uses Hyper-V’s hypervisor to create a secure, isolated environment—a Virtual Secure Mode (VSM) container—where credential material is stored and processed. Even if an attacker gains kernel-level code execution on the host OS, they cannot extract or replay credentials protected by Credential Guard because the hypervisor enforces memory isolation between the VSM container and the main OS.

**Key Components**

- **Virtual Secure Mode (VSM):** A hardware-backed, isolated region of memory created by the hypervisor. VSM runs a separate, minimal instance of Windows (the secure kernel) that hosts the Local Security Authority (LSA) process. This process handles authentication and credential caching entirely within VSM.
- **Hypervisor Code Integrity (HVCI):** Often deployed alongside Credential Guard, HVCI ensures that only signed, validated kernel-mode drivers can run. HVCI and Credential Guard share the same VSM infrastructure, but they are independent features. Enabling both provides defense-in-depth against kernel-level attacks.
- **LSA Isolation:** The standard LSA process (lsass.exe) in the host OS is stripped of credential material. All credential operations are forwarded to the isolated LSA process inside VSM. The host LSA process retains only a communication channel to the isolated LSA.
- **Secure Kernel (SK):** The secure kernel runs in VSM and manages the isolated LSA process. It has its own memory manager and scheduler, separate from the host OS kernel. The SK is signed by Microsoft and is loaded by the hypervisor at boot.
- **Hardware Requirements:** Credential Guard requires a 64-bit CPU with virtualization extensions (Intel VT-x or AMD-V), Second Level Address Translation (SLAT), and Trusted Platform Module (TPM) 2.0 for attestation. UEFI Secure Boot must be enabled. On Windows 10/11, the system must support DMA remapping (VT-d or AMD-Vi) to prevent direct memory access attacks against VSM.

**Real-World Application**

Deploying Credential Guard in an enterprise environment involves three phases: assessment, enablement, and validation.

**Assessment:** Before enabling Credential Guard, verify hardware compatibility. Use the `Get-CimInstance -ClassName Win32_Processor` PowerShell cmdlet to confirm virtualization support. Run `msinfo32.exe` and check for “Virtualization-based security” status. Ensure all drivers are HVCI-compatible; incompatible drivers will cause boot failures. Use the Windows Hardware Lab Kit (HLK) or the Device Guard and Credential Guard hardware readiness tool (available from Microsoft) to scan for issues.

**Enablement:** Credential Guard can be enabled via Group Policy, Intune, or PowerShell. The recommended method is Group Policy: navigate to Computer Configuration > Administrative Templates > System > Device Guard, and set “Turn on Virtualization Based Security” to Enabled. Under “Credential Guard Configuration,” select “Enabled with UEFI lock” to prevent an attacker from disabling it via registry changes. For Intune, use the “Endpoint Protection” profile under “Windows 10 and later” and set “Microsoft Defender Credential Guard” to “Enabled with UEFI lock.” For PowerShell, run:
```
Set-ItemProperty -Path HKLM:\SYSTEM\CurrentControlSet\Control\DeviceGuard -Name EnableVirtualizationBasedSecurity -Value 1
Set-ItemProperty -Path HKLM:\SYSTEM\CurrentControlSet\Control\DeviceGuard -Name RequirePlatformSecurityFeatures -Value 1
Set-ItemProperty -Path HKLM:\SYSTEM\CurrentControlSet\Control\Lsa -Name LsaCfgFlags -Value 1
```
A reboot is required. After reboot, verify with `Get-CimInstance -ClassName Win32_DeviceGuard` and check the “Credential Guard” property.

**Validation:** Confirm that LSA isolation is active. Run `tlist.exe` or `tasklist /m lsaiso.exe` to see the isolated LSA process (lsaiso.exe). Use the Windows Security Center GUI: go to Device Security > Core Isolation > Memory Integrity. If Credential Guard is running, “Memory integrity” will show as “On” (though this specifically indicates HVCI, not Credential Guard). For a definitive check, run `msinfo32.exe` and look for “Virtualization-based security services running” with “Credential Guard” listed.

**Operational Considerations:** Credential Guard breaks some legacy authentication scenarios. It prevents the use of NTLMv1, WDigest, and MS-CHAPv2. It also blocks the use of Kerberos unconstrained delegation and the use of the `runas /netonly` command. Applications that require direct access to credential material (e.g., password managers that inject credentials into logon prompts) will fail. Test all critical applications in a pilot group before broad deployment. For remote desktop scenarios, Credential Guard does not protect credentials used for RDP connections initiated from the protected machine; those credentials are still handled by the host OS.

**Security Relevance**

Credential Guard directly mitigates the most common credential theft techniques used in post-exploitation. Pass-the-Hash (PtH) attacks become ineffective because NTLM hashes are never stored in the host OS memory. Pass-the-Ticket (PtT) attacks are blocked because Kerberos tickets are isolated in VSM. Tools like Mimikatz, which rely on reading lsass.exe memory from the host OS, cannot access the isolated LSA process. Even if an attacker gains SYSTEM privileges, they cannot dump credentials from VSM without breaking the hypervisor isolation—a feat that requires a separate, hardware-level vulnerability.

Credential Guard also reduces the attack surface for lateral movement. If an attacker compromises one workstation, they cannot extract cached domain credentials to pivot to other systems. This containment is critical in environments where users have administrative access to their own machines. Combined with HVCI, Credential Guard provides a layered defense that raises the cost of exploitation significantly.

However, Credential Guard is not a silver bullet. It does not protect against credential theft from memory dumps taken from the host OS before Credential Guard was enabled. It does not prevent keylogging or phishing attacks that capture credentials at the point of entry. It also does not protect against attacks that abuse the communication channel between the host LSA and the isolated LSA (e.g., the “Credential Guard bypass” that uses the LSA notification package). For maximum effectiveness, pair Credential Guard with Windows Defender Application Control (WDAC), BitLocker, and strict user rights assignments.

**Key Takeaways**
- Credential Guard uses hypervisor-based isolation to protect domain credentials from kernel-level theft, rendering Pass-the-Hash and Pass-the-Ticket attacks ineffective.
- Deployment requires hardware virtualization support, UEFI Secure Boot, and compatibility testing with existing drivers and applications; enable with UEFI lock to prevent tampering.
- Credential Guard is a critical component of a defense-in-depth strategy but must be supplemented with phishing-resistant authentication, application control, and network segmentation.
