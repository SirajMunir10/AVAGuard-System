Event logs are a foundational data source for security monitoring, incident response, and compliance auditing. This article focuses on event log ID 7, which is specific to the Windows operating system and indicates that a process has been created.

**Definition**

Event ID 7 in the Windows Security log is generated by the security audit policy subcategory "Audit Process Creation." It records every instance where a new process is started on the system. The event is generated by the kernel and logged by the Local Security Authority Subsystem Service (LSASS). Critically, this event is not enabled by default; it must be explicitly configured via Group Policy or Advanced Audit Policy. Once enabled, it provides a high-fidelity, near-real-time record of process execution, making it indispensable for detecting malicious activity such as code injection, lateral movement, and privilege escalation.

**Key Components**

Each Event ID 7 entry contains several fields that are critical for analysis. The most important are:

- **ProcessID**: The numeric identifier of the newly created process. This is ephemeral and can be reused after the process terminates.
- **Creator ProcessID**: The ProcessID of the parent process that spawned the new process. This establishes the process ancestry.
- **Process Name**: The full path to the executable file of the new process (e.g., `C:\Windows\System32\cmd.exe`).
- **Creator Process Name**: The full path to the executable file of the parent process.
- **User Name**: The account context under which the new process was created. This includes domain and user name (e.g., `DOMAIN\user`).
- **CommandLine**: The complete command line used to launch the process, including arguments. This field is only populated if the "Include command line in process creation events" policy is enabled.
- **Token Elevation Type**: Indicates the type of User Account Control (UAC) elevation applied. Values include:
    - 1: TokenElevationTypeDefault (no elevation)
    - 2: TokenElevationTypeFull (elevated)
    - 3: TokenElevationTypeLimited (non-elevated)
- **Mandatory Label**: The integrity level of the process (e.g., Low, Medium, High, System). This is crucial for detecting processes running at unexpected privilege levels.

**Real-World Application**

Event ID 7 is most effectively used in a Security Information and Event Management (SIEM) or Endpoint Detection and Response (EDR) platform. The following are concrete detection scenarios:

- **Detecting Unauthorized Script Execution**: A rule that triggers when `wscript.exe` or `cscript.exe` is spawned by a Microsoft Office application (e.g., `winword.exe`, `excel.exe`) is a classic indicator of a macro-based attack. The command line field can reveal the specific script path.
- **Identifying Lateral Movement with PsExec**: PsExec creates a service on a remote system. The parent process of the spawned `cmd.exe` or `powershell.exe` will be `services.exe`, and the user name will be the account used for authentication. Correlating this with network logs (Event ID 5156) confirms the remote origin.
- **Detecting Process Hollowing**: A legitimate process (e.g., `svchost.exe`) is created, but its code is replaced. Event ID 7 will show the legitimate process creation, but a subsequent Event ID 4688 (Process Exited) or a memory scan would be needed to confirm the hollowing. However, an anomaly in the parent-child relationship (e.g., `explorer.exe` spawning `rundll32.exe` with no DLL) can be a strong signal.
- **Monitoring for Reconnaissance Tools**: The execution of `whoami.exe`, `netstat.exe`, `ipconfig.exe`, or `tasklist.exe` from an unusual parent process (e.g., `wmiprvse.exe` or a browser) is a common indicator of post-exploitation activity.
- **Tracking Privilege Escalation**: A process created with a `Token Elevation Type` of 2 (Full) from a parent process with a lower integrity level (e.g., Medium to High) indicates a UAC bypass or successful privilege escalation.

**Security Relevance**

Event ID 7 is a cornerstone of a defense-in-depth strategy for several reasons:

- **Visibility into Execution**: It provides the most granular view of what is running on a system. Without it, you are blind to the vast majority of attacker actions that involve executing code.
- **Process Ancestry Analysis**: The parent-child relationship is the single most powerful feature of this event. Attackers rarely follow normal execution chains. A `cmd.exe` spawned by `outlook.exe` is almost always malicious. This allows for behavioral detection that is resistant to signature-based evasion.
- **Command Line Forensics**: When enabled, the command line field captures the exact arguments passed to a process. This is critical for detecting obfuscated PowerShell commands, encoded payloads, or specific file paths used in an attack. It also provides direct evidence for incident response.
- **Compliance Mandates**: Frameworks such as NIST 800-53 (AU-3, AU-12), PCI DSS (Requirement 10.2.2), and CIS Controls (Control 8) require logging of all privileged and user-initiated process executions. Event ID 7 directly satisfies these requirements.
- **Baseline Establishment**: Over time, Event ID 7 data allows you to build a baseline of normal process behavior for each system or user. Deviations from this baseline—such as a user's workstation suddenly running `mimikatz.exe`—become immediately apparent.

**Key Takeaways**

1.  Event ID 7 (Process Creation) must be explicitly enabled via Advanced Audit Policy and the "Include command line" sub-setting to be effective for security monitoring.
2.  The parent-child process relationship and command line arguments are the most valuable fields for detecting malicious activity, enabling behavioral detection that bypasses signature-based tools.
3.  This event is essential for compliance with major security frameworks and provides the foundational visibility required for proactive threat hunting and incident response.
