Snort rules are the core logic that drives the Snort intrusion detection and prevention system (IDS/IPS). A Snort rule is a single, declarative statement that defines a specific pattern of network traffic to detect or block. Each rule is processed against every packet traversing the monitored interface. Understanding the structure and syntax of these rules is essential for creating custom detections, tuning false positives, and maintaining an effective security posture.

**Key Components of a Snort Rule**

Every Snort rule consists of two primary sections: the rule header and the rule options. The rule header defines the action, protocol, source and destination IP addresses, and ports. The rule options define the specific content or conditions to match.

**Rule Header Structure:**
- **Action:** The action to take when a match occurs. Common actions include:
    - `alert`: Generate an alert and log the packet.
    - `log`: Log the packet without alerting.
    - `pass`: Ignore the packet (used for whitelisting).
    - `drop` (inline mode): Block the packet and log it.
    - `reject` (inline mode): Block the packet, log it, and send a TCP RST or ICMP unreachable.
- **Protocol:** The IP protocol to inspect (e.g., `tcp`, `udp`, `icmp`, `ip`).
- **Source IP and Port:** The originating address and port. Can be `any`, a specific IP, a CIDR block (e.g., `192.168.1.0/24`), or a variable (e.g., `$HOME_NET`). The `!` operator negates the match (e.g., `!$HOME_NET`).
- **Direction Operator:** Indicates the flow of traffic. `->` means from source to destination. `<>` means bidirectional.
- **Destination IP and Port:** The target address and port, using the same syntax as source.

**Rule Options (within parentheses):**
Options are separated by semicolons and provide the detection logic. Key options include:
- `msg:"description"`: A human-readable alert message.
- `content:"pattern"`: Matches a specific byte sequence in the packet payload. Can be modified with modifiers like `nocase`, `rawbytes`, `depth`, `offset`, `within`, and `distance`.
- `pcre:"/regex/"`: Matches using Perl-compatible regular expressions. More flexible but computationally expensive.
- `sid:unique_number`: A unique Snort rule ID. Must be unique across all rules.
- `rev:number`: The revision number of the rule, incremented with each update.
- `classtype:category`: Maps the rule to a predefined attack category (e.g., `attempted-admin`, `trojan-activity`).
- `priority:number`: Sets the priority level (1 highest, 4 lowest).
- `flow:direction,state`: Matches based on TCP stream state (e.g., `to_client`, `from_server`, `established`, `stateless`).
- `byte_test:bytes,offset,operator,value`: Extracts a number of bytes from the payload and compares them using an operator (e.g., `=`, `>`, `<`).
- `threshold:type,count,seconds`: Limits the number of alerts generated by this rule within a time window to reduce noise.

**Real-World Application: Writing a Custom Rule**

Consider a scenario where you need to detect an internal host scanning for open SMB ports (TCP 445) across a subnet. A basic rule might look like this:

`alert tcp $HOME_NET any -> $HOME_NET 445 (msg:"Potential SMB port scan detected"; flow:to_server; threshold:type both, track by_src, count 10, seconds 60; sid:1000001; rev:1;)`

This rule alerts when any host in `$HOME_NET` sends more than 10 packets to TCP port 445 on any other host in `$HOME_NET` within 60 seconds. The `threshold` option prevents alert fatigue by aggregating events.

For a more precise detection, such as identifying a specific exploit payload, you would use `content` and `pcre` options. For example, to detect an attempt to exploit EternalBlue (MS17-010):

`alert tcp $EXTERNAL_NET any -> $HOME_NET 445 (msg:"ETERNALBLUE exploit attempt"; flow:to_server,established; content:"|00 00 00 31 ff|"; depth:4; offset:20; content:"|ff|"; distance:0; within:1; sid:1000002; rev:1;)`

This rule matches a specific byte sequence at a known offset within an SMB session, indicating the exploit payload. The `depth` and `offset` modifiers constrain the search to a specific location in the packet, reducing false positives.

When deploying custom rules, always test them in a lab environment using a tool like `tcpreplay` or by generating benign traffic. Use Snort’s `-T` (test mode) flag to check for syntax errors before loading rules into production.

**Security Relevance**

Properly written Snort rules are the foundation of network-based threat detection. They enable security teams to:
- **Detect known attacks:** Signature-based detection for malware, exploits, and policy violations.
- **Identify anomalous behavior:** Rules can flag traffic patterns that deviate from baselines, such as excessive outbound DNS queries or unusual port combinations.
- **Enforce compliance:** Rules can monitor for unencrypted protocols (e.g., FTP, Telnet) or sensitive data exfiltration attempts.
- **Reduce false positives:** Well-crafted rules with precise content matching, flow state awareness, and thresholding minimize noise, allowing analysts to focus on genuine threats.
- **Support incident response:** Alert messages and rule metadata provide context for triage and investigation.

Poorly written rules, however, can degrade performance, generate excessive false positives, or miss critical threats. Overly broad rules (e.g., using `content:"|00|"` without constraints) can cause Snort to consume excessive CPU and memory. Conversely, rules that are too narrow may fail to detect variants of an attack.

**Key Takeaways**
1.  A Snort rule is composed of a header (action, protocol, addresses, ports) and options (content matching, flow state, thresholds) that together define a precise detection signature.
2.  Use modifiers like `depth`, `offset`, `within`, and `distance` to constrain content searches, and apply `threshold` or `flow` options to reduce false positives and manage alert volume.
3.  Always validate custom rules with Snort’s `-T` flag and test against representative traffic before deployment to ensure accuracy and performance.
