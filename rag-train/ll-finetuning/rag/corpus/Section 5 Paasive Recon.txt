================================================================================
SECTION 5: PASSIVE RECONNAISSANCE
================================================================================

Passive reconnaissance gathers intelligence about a target without sending any packets directly to the target's systems. Because no direct interaction occurs, passive recon is legal and undetectable regardless of whether formal authorization has been granted. It is always the correct starting point for any external assessment. The goal is to build a detailed picture of the organization's attack surface using only publicly available sources before any authorized active testing begins.

WHOIS and DNS enumeration are foundational passive recon techniques. WHOIS lookups reveal registrant information, registration dates, name servers, and sometimes direct contact details for an organization. DNS enumeration using tools like dnsrecon and amass identifies subdomains, mail exchanger records, text records containing SPF and DKIM policies, and other DNS entries that reveal the structure of the organization's infrastructure. DNS record types of particular interest include A records mapping hostnames to IP addresses, MX records revealing email infrastructure, TXT records that may contain API verification tokens, and CNAME records pointing to third-party services.

Certificate transparency logs are a valuable source of subdomain intelligence. Services like crt.sh and Censys index every TLS certificate issued by public certificate authorities. By searching for certificates issued to a target domain, a tester can discover subdomains that may not be discoverable through brute-force DNS enumeration. Wildcard certificates and certificates for internal-sounding hostnames such as dev.example.com, staging.example.com, or vpn.example.com are particularly useful leads for further investigation.

Search engine dorking uses advanced query operators in search engines like Google to discover sensitive information indexed from target websites. Common dorks include searching for site:example.com to enumerate all indexed pages, filetype:pdf site:example.com to find downloadable documents, and intitle:index.of site:example.com to locate directory listings. The Shodan and Censys search engines index internet-connected devices and can reveal open ports, service versions, SSL certificate details, and configuration files exposed on the target's infrastructure.

LinkedIn and professional social media platforms are essential for building an organizational chart and identifying high-value targets for social engineering. Job postings are particularly valuable as they often reveal the exact technology stack in use, such as specific SIEM products, cloud providers, or development frameworks. Employee names and roles discovered through LinkedIn can be used to generate plausible email addresses for phishing campaigns using common naming convention patterns like first.last@example.com.

Credential breach databases such as HaveIBeenPwned and commercial services like DeHashed allow testers to identify whether any corporate email addresses appear in publicly known data breaches. Breached credentials are commonly reused across services and can enable immediate access through credential stuffing against VPN portals, cloud consoles, or web applications without any exploitation required.

GitHub and public code repositories are a rich source of inadvertently exposed secrets. Developers frequently commit API keys, database connection strings, private keys, and internal hostnames to public repositories. Searches using GitHub's code search for the organization's domain name, product names, or common secret patterns can surface credentials and configuration details that were never intended to be public. Tools like TruffleHog and GitLeaks can automate this search at scale.
